# 내부고발자 매니페스트 봉인 키트

**내부고발자와 공중 사이의 중개자들에 의해 증거가 묻히거나, 편집되거나, 분쟁화되는** 특정한 실패 양식에 직면한 내부고발자를 위한 턴키 myriam-kit 예시입니다. 이 키트가 방어하는 표준적인 역사적 사례:

- Boeing 737 MAX (내부 안전 보고서가 평가절하되거나 묻힘)
- Theranos (수년간 검사 결과 우려가 억압됨)
- FDA 검사관 보고서가 제출 이후 편집됨
- Frances Haugen / Facebook (봉인된 매니페스트가 있었다면 문서가 진본이며 주장된 시점에 존재했음을 증명할 수 있었음)

이 키트가 (프로토콜 용어로) 방어하는 것:

- [FACT 05](facts/05_threat_model.txt)의 T1–T3: 문서 내용에 대한 공개적 분쟁, 문서가 주장된 시점에 봉인되지 않았다는 주장, 소급적 조작 주장.
- T9: 제3자가 변조된 사본을 기자들에게 전달하는 것.

이 키트가 **방어하지 않는** 것:

- 내부고발자에 대한 개인적 보복 (T7).
- 내부고발자에게 철회를 압박하는 것 (T4, FACT 07이 침묵 철회를 공개적으로 탐지 가능하게 함으로써 부분적으로 완화).
- 공개 그 자체에 대한 내부고발자에 대한 법적 조치 (T5).
- 내부고발자의 인격에 대한 폄훼 (T8).
- 중개자가 침묵으로 사안을 묻어버리는 것 (T10).

이 키트의 사용을 고려하는 내부고발자라면, **먼저 이 README 전체를 읽으십시오**. 프로토콜은 도구입니다. 프로토콜은 변호사, 기자, 규제 당국, 또는 신중한 운영 보안(operational security)의 대체물이 아닙니다.

## 이 키트가 산출하는 것

증거의 (증거 자체가 아니라) 암호학적으로 봉인된 *매니페스트*로, 다음을 증명합니다:

1. 각 증거 파일의 특정 바이트가 봉인 시점에 존재했다.
2. 봉인이 특정 시점 또는 그 이전에 발생했다(Bitcoin OTS 증명을 통해).
3. 출처, 제소 사실, 위협 모델, 수신자 프로토콜, 침묵 철회 금지 약속이 모두 함께 선언되었다.

기반 문서는 수신자에게 전송될 때까지 여러분의 소유에 머뭅니다. 수신자는 자신이 받은 바이트가 여러분이 봉인한 바이트와 동일한지 검증합니다.

## 언제 이 키트를 사용하는가

증거를 기자, 규제 당국, 또는 변호사에게 전송하기 **전에** 매니페스트 봉인을 고려해야 하는 경우는 다음이 모두 참일 때입니다:

- 바이트 단위로 진본임을 보존하고자 하는 디지털 형태의 증거(PDF, 이메일, 이미지 등)를 보유하고 있다.
- 중개자들이 증거를 변경, 분쟁화, 또는 묻을 수 있다고 예상된다.
- 전송 시점까지 기반 문서를 자신의 소유로 유지할 수 있다.
- 문서 내용을 노출하지 **않은 채** 매니페스트를 공개할 수 있다(해시만 공개).
- 내부고발자로 공개적으로 식별될 위험(서명 주체 필드)을 저울질했다. 익명일 경우, 모든 관련 공표물 전반에 걸쳐 일관된 가명을 사용하십시오.
- 자신의 관할권 내 내부고발자 보호법에 대해 변호사와 상담했다.

## 이 키트를 사용하지 **않아야** 할 때

다음의 경우 이 키트를 사용하지 마십시오:

- 문서 자체에 공개 해시로 만들어서는 안 되는 제3자(환자, 고객, 직원)의 개인정보가 포함되어 있는 경우. 원본이 아니라 비식별 처리된 버전을 해시하십시오.
- 매니페스트 공개라는 행위 자체가 동의하지 않은 방식으로 내부고발자를 식별하게 되는 경우.
- 아직 변호사와 상담하지 않은 경우. 프로토콜은 도구일 뿐이며, 내부고발자 보호법은 관할권별로 다르고 프로토콜 외부에 존재합니다.
- 단 하나의 제소라도 나중에 철회하고 싶을 수 있는 경우. 봉인은 영구적이며, 새로 봉인된 정정만이 솔직하게 수정하는 유일한 방법입니다.
- 강압 또는 시간 압박 하에 작업하는 경우. 프로토콜은 강압을 증폭합니다. 거부 게이트(myriam-kit `SPEC §8`)는 발행자에게뿐 아니라 내부고발자에게도 적용됩니다.

## 실제 공개를 위해 이 키트를 포크하는 방법

```bash
# 1. 이 예시를 새 디렉터리로 복사
cp -r ~/Genesis/myriam-kit/examples/whistleblower-manifest ~/my-disclosure
cd ~/my-disclosure

# 2. 가상의 콘텐츠를 제거하고 실제 사실로 교체.
# facts/ 아래의 각 파일은 하나의 사실입니다. 각각을 읽고 교체하십시오.
$EDITOR facts/01_example_notice.txt   # 실제 표지 진술로 변경
$EDITOR facts/02_provenance.txt       # 증거의 실제 출처
$EDITOR facts/03_evidence_manifest.txt # 실제 문서를 해시하여 목록에 기재
$EDITOR facts/04_allegation_summary.txt # 실제 제소 사실
$EDITOR facts/05_threat_model.txt     # 예상되는 실제 반응
$EDITOR facts/06_recipient_protocol.txt # 수신자에 맞춰 조정
$EDITOR facts/07_no_retraction_commitment.txt # 철회 금지 선언
$EDITOR facts/08_signing_principal.txt # 실제 서명 신원 (또는 가명)

# 3. site/index.html을 실제 사실에 맞게 갱신.
# seal.sh 스크립트는 HTML을 자동 채우기 하지 않으며, 수동 편집은
# 사람의 검토 단계입니다.
$EDITOR site/index.html

# 4. 봉인.
bash build/seal.sh
# 출력: merkle_root.txt, merkle_root.txt.ots, IPFS CID

# 5. 로컬 검증.
python3 build/verify.py

# 6. 매니페스트(문서가 아님)를 다음 중 하나에 공표:
#    - 자신이 통제하는 정적 호스트 (Cloudflare Pages, GitHub Pages 등)
#    - 출력된 CID로 IPFS 네트워크
#    - 둘 다 (권장)

# 7. 별도로, 기반 문서를 선택한 안전한 채널 (Signal,
#    SecureDrop 등)을 통해 수신자에게 전송.
#    수신자는 매니페스트와 파일 해시를 대조하여 검증한다.
```

## 실제 증거 파일의 해시 계산

문서 파일을 SHA-256으로 해시하려면:

```bash
shasum -a 256 my_document.pdf
```

처음 64자리 16진 문자가 SHA-256입니다. 그 값을 매니페스트에 기재하십시오.

여러 파일을 한 번에 해시할 수도 있습니다:

```bash
shasum -a 256 *.pdf *.eml *.jpg > MANIFEST.txt
```

이것이 `facts/03_evidence_manifest.txt`를 위한 데이터입니다.

## 운영 보안 관련 메모

MYRIAM 프로토콜은 운영 보안(operational security)을 제공하지 **않습니다**. 제공하는 것은 *증거적 영속성*입니다. 여전히 필요한 것:

- 변호사에 대한 안전한 통신 채널 ([Signal](https://signal.org/), 대면, 또는 변호사가 확립한 안전한 채널)
- 기반 문서의 안전한 저장 위치 (저장 시 암호화되어 있으며, 고용주가 접근할 수 있는 클라우드 서비스가 아닐 것)
- 봉인을 위한 깨끗한 기기 (고용주 계정과 연관되지 않은 개인 노트북, 가급적 이 용도로만 사용되는 기기)
- 공표를 위한 깨끗한 네트워크 (고용주의 네트워크가 아니어야 하고, 고용주가 ISP를 소환할 가능성이 있다면 자택 IP도 아닐 것)

내부고발자를 위한 운영 보안 권장 읽을거리:

- [The Intercept의 SecureDrop 문서](https://securedrop.org/)
- [Freedom of the Press Foundation 자료](https://freedom.press/)
- [EFF의 감시 자기방어 가이드](https://ssd.eff.org/)

봉인 프로세스를 시작하기 **전에** 그것들을 참조하십시오. 프로토콜은 프로토콜이 호출되기 이전에 내려진 잘못된 운영 보안 결정을 방어하지 못합니다.

## 수신자가 알아야 할 사항

봉인된 매니페스트와 함께 이 README를 수신자에게 송부하십시오. 매니페스트 자체에 수신자 프로토콜(FACT 06)이 포함되어 있지만, README는 수신자의 역할을 확장합니다:

- **기자**: 먼저 매니페스트 자체의 무결성을 검증하십시오. 그런 다음 별도의 안전한 채널을 통해 문서를 요청하십시오. 공표 전에, 각 문서의 SHA-256이 매니페스트와 일치하는지 검증하십시오. 내부고발자의 동의 없이 문서를 공표하지 **마십시오**. 봉인된 매니페스트의 존재 및 자신의 검증 결론은 공표하십시오.
- **규제 당국**: 동일한 검증 프로세스. 봉인된 매니페스트 + 일치하는 문서는 공식적 보관 사슬 수집을 제외하면 입수 가능한 가장 강력한 증거 형식입니다.
- **변호사**: 봉인된 매니페스트는 주장된 시점에 문서가 존재했다는 점에 대해 포렌식 정박을 제공합니다. 그것이 특정 관할권의 증거 규칙을 충족하는지는 귀하의 판단입니다.

## 이 키트가 **아닌** 것

- 변호사 상담의 대체물이 아닙니다.
- 누군가가 귀하의 공개에 따라 행동할 것이라는 보장이 아닙니다.
- 보복에 대한 보호가 아닙니다.
- 문서 내용에 대한 인증이 아닙니다(봉인 시점의 바이트만 인증).
- 누군가와 익명으로 소통하는 방법이 아닙니다 — 봉인은 공개됩니다. 서명 주체 이름이 실명이라면, 귀하는 식별됩니다.

## 맺음말

봉인된 매니페스트는 작은 것입니다. 그것은 특정 시점에 특정 형태로 더 큰 바이트 스트림 집합이 존재했음을 증명하는 하나의 바이트 스트림입니다. 그 작음이 곧 그것의 강점입니다. 그것은 단 하나의 일을 하며, Bitcoin 블록체인의 수명에 걸쳐 그 하나의 일을 합니다.

내부고발자가 되는 것을 고려하고 있어 이 글을 읽고 있다면, 부탁드립니다: 먼저 변호사와 상담하십시오. 위의 운영 보안 가이드를 활용하십시오. 그런 다음, 그리고 모든 상류의 고려사항이 정돈된 경우에 한해, 이 키트를 공개의 암호학적 척추로 고려하십시오. 암호는 쉬운 부분입니다. 암호의 상류에 있는 모든 것이 실제 위험이 자리한 곳입니다.
