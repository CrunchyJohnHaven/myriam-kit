# 举报人清单密封工具包

一个面向举报人的 myriam-kit 即开即用范例,针对的是这样一种特定失败模式:
**证据被举报人与公众之间的中间人埋没、编辑或质疑**。该工具包所防御的
经典历史案例包括:

- 波音 737 MAX(内部安全报告被淡化/埋没)
- Theranos(实验室检测的疑虑被压制多年)
- FDA 检查员报告在提交后遭到编辑
- Frances Haugen / Facebook(若有密封清单即可证明文档真实
  且在所声称时间存在)

该工具包所防御的内容(以协议术语描述):

- [FACT 05](facts/05_threat_model.txt) 中的 T1–T3:对文档内容的公开
  争议、声称文档未在所述时间被密封、声称属于事后伪造。
- T9:第三方将经过篡改的副本交给记者。

该工具包**不**防御的内容:

- 对举报人的个人报复(T7)。
- 迫使举报人撤回的压力(T4,FACT 07 通过使悄然撤回可被公开
  检测而部分缓解)。
- 因披露行为本身而对举报人提起的法律诉讼(T5)。
- 对举报人品格的抹黑(T8)。
- 中间人悄然放弃报道(T10)。

如果你是一位正在考虑使用此工具包的举报人,**请先完整阅读本 README**。
协议只是一种工具。协议不能替代法律顾问、记者、监管者或审慎的运营安全。

## 此工具包产出什么

一份证据的密码学密封*清单*(而非证据本身),用以证明:

1. 每个证据文件的具体字节在密封时刻就已存在。
2. 密封发生于特定时间或之前(通过 Bitcoin OTS 见证)。
3. 来源、指控、威胁模型、接收方协议以及不得悄然撤回的承诺
   是一并声明的。

底层文档在你将其发送给接收方之前仍由你保管。接收方验证他们收到的
字节与你所密封的相同。

## 何时使用此工具包

在向记者、监管者或律师传递证据**之前**,如果以下条件全部成立,你应
考虑密封一份清单:

- 你拥有数字形式的证据(PDF、电子邮件、图像等),希望保留为
  字节级真实的形式。
- 你预期中间人可能会篡改、质疑或埋没证据。
- 在传递之前,你可以保留对底层文档的占有。
- 你可以公开发布该清单,而**无需**透露文档内容(仅哈希是公开的)。
- 你已权衡了作为举报人被公开识别的风险(即签名主体字段)。如需
  匿名,请在所有相关发布中一致使用化名。
- 你已就所在司法管辖区的举报人保护法咨询律师。

## 何时**不**使用此工具包

如有以下情形,请勿使用此工具包:

- 文档本身包含第三方(患者、客户、雇员)的个人信息,这些信息
  不应被公开可哈希化。请对脱敏后的版本进行哈希,而非原始文档。
- 发布清单这一行为本身会以举报人未曾同意的方式将其暴露身份。
- 你尚未咨询律师。协议只是工具;举报人保护法因司法管辖区而异,
  存在于协议之外。
- 你日后可能希望撤回任何一项指控。密封是永久性的;新的密封更正
  是唯一诚实的修订方式。
- 你正处于胁迫或时间压力之下。该协议会放大胁迫;拒绝门
  (myriam-kit `SPEC §8`)同样适用于举报人和发布者。

## 如何为真实披露派生该工具包

```bash
# 1. 将此范例复制到一个新目录
cp -r ~/Genesis/myriam-kit/examples/whistleblower-manifest ~/my-disclosure
cd ~/my-disclosure

# 2. 删除虚构内容;替换为你的真实事实。
# facts/ 下的每个文件即一项事实。请逐一阅读并替换。
$EDITOR facts/01_example_notice.txt   # 改为真实的封面声明
$EDITOR facts/02_provenance.txt       # 你的证据的真实来源
$EDITOR facts/03_evidence_manifest.txt # 对你的真实文档进行哈希并列出
$EDITOR facts/04_allegation_summary.txt # 你的真实指控
$EDITOR facts/05_threat_model.txt     # 你预期的真实应对反应
$EDITOR facts/06_recipient_protocol.txt # 根据你的接收方调整
$EDITOR facts/07_no_retraction_commitment.txt # 你的不得撤回声明
$EDITOR facts/08_signing_principal.txt # 你真实的签名身份(或化名)

# 3. 更新 site/index.html 以反映真实事实。
# seal.sh 脚本不会自动填充 HTML;手动编辑即为人工审阅步骤。
$EDITOR site/index.html

# 4. 密封。
bash build/seal.sh
# 输出:merkle_root.txt、merkle_root.txt.ots、IPFS CID

# 5. 本地验证。
python3 build/verify.py

# 6. 将清单(而非文档)发布至以下任一渠道:
#    - 你自行控制的静态托管(Cloudflare Pages、GitHub Pages 等)
#    - IPFS 网络上的所打印 CID
#    - 两者(建议)

# 7. 另行通过你所选的安全通道(Signal、SecureDrop 等),将底层
#    文档传递给接收方。接收方依据你的清单核验文件哈希。
```

## 计算真实证据文件的哈希

使用 SHA-256 对文档文件进行哈希:

```bash
shasum -a 256 my_document.pdf
```

前 64 个十六进制字符即为 SHA-256。将其填入清单中。

你可以一次性对多个文件进行哈希:

```bash
shasum -a 256 *.pdf *.eml *.jpg > MANIFEST.txt
```

这便给你提供了 `facts/03_evidence_manifest.txt` 所需的数据。

## 运营安全注意事项

MYRIAM 协议**不**提供运营安全。它提供的是*证据的持久性*。你仍然需要:

- 通往律师的安全通信通道
  ([Signal](https://signal.org/)、面对面,或律师既有的安全通道)
- 用于底层文档的安全存储位置(静态加密,不要放在你雇主可访问的
  云服务上)
- 用于密封的干净设备(一台不与雇主账户关联的个人笔记本电脑;
  最好仅用于此用途)
- 用于发布的干净网络(不是你雇主的网络;若你雇主可能向你的 ISP
  发出传票,也不要使用你家中的 IP)

针对举报人运营安全的推荐阅读:

- [The Intercept 的 SecureDrop 文档](https://securedrop.org/)
- [新闻自由基金会的资源](https://freedom.press/)
- [EFF 的监控自卫指南](https://ssd.eff.org/)

在你启动密封流程**之前**请先咨询这些资源。协议无法防御在协议被调用
之前所做出的糟糕运营安全选择。

## 接收方应当了解的内容

请将本 README 连同密封清单一并发送给接收方。清单本身包含接收方协议
(FACT 06),但 README 进一步阐述了接收方角色:

- **记者**:首先验证清单自身的完整性。然后通过另一条安全通道
  请求文档。在发表之前,核验每份文档的 SHA-256 与清单一致。
  未经举报人同意请勿发表文档本身;但应当发表密封清单的存在
  以及你的验证结论。
- **监管者**:相同的验证流程。密封清单加上相匹配的文档是
  正式监管链取证之外可获得的最强证据形式。
- **律师**:密封清单为文档在所述时间已存在提供取证锚定。
  其是否满足任一特定司法管辖区的证据规则,需由你判定。

## 此工具包**不**是什么

- 不是法律顾问的替代品。
- 不保证任何人会就你的披露采取行动。
- 不能防御报复。
- 不对文档内容进行认证(仅认证密封时刻的字节)。
- 不是与任何人匿名通信的方式 —— 密封是公开的;若你的签名主体
  名称真实,则你已被识别。

## 结语

一份密封清单是一个微小之物。它是一段字节流,证明在某一特定时刻、
以某一特定形态存在着更大的一组字节流。这种微小正是它的力量所在:
它只做一件事,并能在 Bitcoin 区块链的整个生命周期内坚持做这一件事。

如果你正在阅读本文,因为你正在考虑成为一名举报人,请:先与律师交谈。
使用上述运营安全指南。然后,当且仅当所有这些上游考量都已就绪时,
再将此工具包视为你披露行动的密码学骨干。密码学是容易的部分。密码学
上游的一切,才是真实风险所在之处。
