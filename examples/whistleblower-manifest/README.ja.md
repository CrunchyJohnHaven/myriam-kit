# 内部告発者マニフェスト封緘キット

**内部告発者と公衆の間に位置する仲介者によって、証拠が埋もれたり、編集されたり、争われたりする**という固有の失敗モードに直面する内部告発者向けの、myriam-kit によるターンキー型のサンプルです。本キットが防御対象とする典型的な歴史的事例:

- Boeing 737 MAX(内部安全報告書が軽視・隠蔽された)
- Theranos(検査結果に関する懸念が何年も抑え込まれた)
- FDA 査察官の報告書が提出後に編集された
- Frances Haugen / Facebook(封緘されたマニフェストがあれば、文書が真正であり主張された時点に存在していたことを証明できたはず)

このキットが防御するもの(プロトコルの用語で):

- [FACT 05](facts/05_threat_model.txt) における T1–T3: 文書内容に対する公的な異議申し立て、文書が主張された時点では封緘されていなかったとの主張、遡及的捏造の主張。
- T9: 第三者がジャーナリストに改竄されたコピーを渡すこと。

このキットが防御**しない**もの:

- 内部告発者本人への報復(T7)。
- 撤回するよう内部告発者にかけられる圧力(T4、ただし FACT 07 によって沈黙裡の撤回を公的に検出可能にすることで部分的に緩和)。
- 開示自体に対する内部告発者への法的措置(T5)。
- 内部告発者の人格に対する信用毀損(T8)。
- 仲介者が沈黙裡に話を握り潰すこと(T10)。

このキットの使用を検討している内部告発者の方は、**まずこの README を最後まで読んでください**。プロトコルは一つの道具です。プロトコルは、弁護士、ジャーナリスト、規制当局、あるいは慎重な運用上の安全対策(オペレーショナル・セキュリティ)の代替にはなりません。

## このキットが生成するもの

証拠ファイルの暗号学的に封緘された*マニフェスト*(証拠そのものではない)であり、以下を証明します:

1. 各証拠ファイルの特定のバイト列が封緘時点で存在していたこと。
2. 封緘が特定の時刻またはそれ以前に行われたこと(Bitcoin OTS 証明による)。
3. 出所、申し立て、脅威モデル、受領者プロトコル、および沈黙裡の撤回を行わないというコミットメントが、すべて同時に宣言されたこと。

基礎となる文書そのものは、受領者に送付されるまであなたの手元にとどまります。受領者は、自分が受け取ったバイト列があなたが封緘したものと同一であることを検証します。

## このキットを使用すべきとき

ジャーナリスト、規制当局、または弁護士に証拠を伝達する**前に**マニフェストの封緘を検討すべきなのは、以下のすべてが当てはまる場合です:

- バイトレベルで真正性を保全したいデジタル形式の証拠(PDF、メール、画像など)を保有している。
- 仲介者が証拠を改竄、紛争化、または隠蔽するおそれがあると見込んでいる。
- 伝達まで基礎となる文書を自分の手元に保持できる。
- 文書の内容を明かすことなく(公開されるのはハッシュのみ)、マニフェストを公開できる。
- 内部告発者として公的に特定されうるリスク(署名主体フィールド)を考慮済みである。匿名の場合は、関連する全公開物にわたって一貫した仮名を使用してください。
- ご自身の管轄における内部告発者保護法について、弁護士に相談済みである。

## このキットを使用すべきでないとき

以下の場合は、このキットを使用しないでください:

- 文書自体に、公的にハッシュ化すべきでない第三者(患者、顧客、従業員)の個人情報が含まれている場合。原本ではなく、伏字処理した版をハッシュしてください。
- マニフェスト公開という行為そのものが、本人が同意していない形で内部告発者を特定してしまう場合。
- 弁護士にまだ相談していない場合。プロトコルは道具にすぎず、内部告発者保護法は管轄ごとに異なり、プロトコルの外側に存在します。
- いずれか一つの申し立てを後日撤回したくなる可能性がある場合。封緘は永続的であり、誠実に修正する唯一の方法は新たな封緘済み訂正を発行することです。
- 強制下や時間的圧力のもとで実行している場合。プロトコルは強制を増幅します。拒否ゲート(myriam-kit `SPEC §8`)は、発行者と同様に内部告発者にも適用されます。

## 実際の開示のためにこのキットをフォークする方法

```bash
# 1. この例を新しいディレクトリにコピー
cp -r ~/Genesis/myriam-kit/examples/whistleblower-manifest ~/my-disclosure
cd ~/my-disclosure

# 2. 架空の内容を削除し、実際の事実に置き換える。
# facts/ 配下の各ファイルは 1 つの事実です。各々を読んで置き換えます。
$EDITOR facts/01_example_notice.txt   # 実際の表紙声明に変更
$EDITOR facts/02_provenance.txt       # 証拠の実際の出所
$EDITOR facts/03_evidence_manifest.txt # 実文書をハッシュ化し、一覧に記載
$EDITOR facts/04_allegation_summary.txt # 実際の申し立て
$EDITOR facts/05_threat_model.txt     # 想定される実際の反応
$EDITOR facts/06_recipient_protocol.txt # 受領者に合わせて調整
$EDITOR facts/07_no_retraction_commitment.txt # 撤回しないことの宣言
$EDITOR facts/08_signing_principal.txt # 実際の署名アイデンティティ(または仮名)

# 3. site/index.html を実際の事実に合わせて更新。
# seal.sh スクリプトは HTML を自動補完しません。この手動編集は
# 人による確認(ヒューマン・レビュー)のステップです。
$EDITOR site/index.html

# 4. 封緘。
bash build/seal.sh
# 出力: merkle_root.txt, merkle_root.txt.ots, IPFS CID

# 5. ローカルで検証。
python3 build/verify.py

# 6. マニフェスト(文書そのものではない)を以下のいずれかに公開:
#    - 自分が管理する静的ホスト(Cloudflare Pages、GitHub Pages 等)
#    - 出力された CID で IPFS ネットワーク
#    - 両方(推奨)

# 7. 別途、基礎となる文書を、お好みの安全なチャネル
#    (Signal、SecureDrop 等)で受領者に伝達する。
#    受領者は、マニフェストに照らしてファイルハッシュを検証する。
```

## 実際の証拠ファイルのハッシュを計算する

文書ファイルを SHA-256 でハッシュするには:

```bash
shasum -a 256 my_document.pdf
```

最初の 64 文字の 16 進数が SHA-256 です。これをマニフェストに記載します。

複数のファイルを一度にハッシュすることもできます:

```bash
shasum -a 256 *.pdf *.eml *.jpg > MANIFEST.txt
```

これが `facts/03_evidence_manifest.txt` のデータとなります。

## 運用上の安全対策(オペセック)に関する注記

MYRIAM プロトコルは運用上の安全対策(オペレーショナル・セキュリティ)を提供しません。提供するのは*証拠の永続性*です。次のものは依然として必要です:

- 弁護士との安全な通信チャネル([Signal](https://signal.org/)、対面、または弁護士が確立している安全なチャネル)
- 基礎となる文書の安全な保管場所(保存時に暗号化されており、雇用主がアクセスできるクラウドサービス上ではないこと)
- 封緘のためのクリーンなデバイス(雇用主のアカウントと紐付いていない個人ラップトップ、できればこの用途専用に使われているもの)
- 公開のためのクリーンなネットワーク(雇用主のネットワークではなく、雇用主が ISP に召喚状を出す可能性があるなら自宅 IP でもないこと)

内部告発者向けの運用上の安全対策に関する推奨資料:

- [The Intercept による SecureDrop ドキュメント](https://securedrop.org/)
- [Freedom of the Press Foundation のリソース](https://freedom.press/)
- [EFF のサーベイランス自衛ガイド](https://ssd.eff.org/)

封緘プロセスを開始する**前に**、これらを必ず参照してください。プロトコルは、プロトコル呼び出し前に下された不適切なオペセックの選択を防御するものではありません。

## 受領者が知っておくべきこと

封緘済みマニフェストとともに、この README を受領者に送ってください。マニフェスト自体にも受領者プロトコル(FACT 06)が含まれていますが、README は受領者の役割を補足します:

- **ジャーナリスト**: まずマニフェスト自体の整合性を検証してください。続いて別個の安全なチャネルを通じて文書を要求してください。公開前に、各文書の SHA-256 がマニフェストと一致することを検証してください。内部告発者の同意なしに文書を公開してはなりません。封緘されたマニフェストの存在と、ご自身の検証結論については公開してください。
- **規制当局**: 同じ検証プロセスを行ってください。封緘済みマニフェストと一致する文書は、正式な証拠保全手続による収集を除けば、入手しうる最も強力な証拠形式です。
- **弁護士**: 封緘済みマニフェストは、文書が主張された時点に存在していたことに対するフォレンジック・アンカーを提供します。それが特定の管轄の証拠規則を満たすかどうかは、あなたの判断によります。

## このキットが**ない**もの

- 弁護士による助言の代替ではありません。
- 誰かが開示に対して行動を起こすことを保証するものではありません。
- 報復に対する保護ではありません。
- 文書の内容に対する認証ではありません(封緘時点のバイト列のみを認証します)。
- 誰かと匿名で通信する手段ではありません — 封緘は公開されます。署名主体の名前が実名であれば、あなたは特定されます。

## 結びに

封緘済みマニフェストは小さなものです。それは、特定の時点に特定の形式で、より大きなバイト列の集合が存在していたことを証明する、一つのバイト列です。その小ささこそが強みです。それは一つのことだけを行い、その一つのことを Bitcoin ブロックチェーンの寿命にわたって行います。

もし内部告発者になることを検討していてこれを読んでいるなら、まず弁護士に相談してください。上記の運用上の安全対策ガイドを参照してください。そのうえで、これらすべての上流の検討事項が整っている場合に限り、本キットを開示の暗号学的バックボーンとしてご検討ください。暗号は簡単な部分です。暗号より上流のすべての場所こそ、真のリスクが存在する場所です。
