# व्हिसलब्लोअर मेनिफेस्ट सील किट

व्हिसलब्लोअरों (मुखबिरों) के लिए एक तैयार-उपयोग वाला myriam-kit उदाहरण, जो उस विशेष विफलता मोड का सामना करते हैं जहाँ **व्हिसलब्लोअर और जनता के बीच मध्यस्थों द्वारा साक्ष्य दबाया, संपादित, या विवादित किया जाता है**। वे विहित ऐतिहासिक मामले जिनके विरुद्ध यह किट रक्षा करती है:

- Boeing 737 MAX (आंतरिक सुरक्षा रिपोर्टों की अनदेखी की गई / दबा दी गईं)
- Theranos (प्रयोगशाला-परीक्षण की चिंताओं को वर्षों तक दबाया गया)
- FDA निरीक्षक रिपोर्टें जो प्रस्तुति के बाद संपादित की गईं
- Frances Haugen / Facebook (एक सीलबंद मेनिफेस्ट यह सिद्ध कर देता कि दस्तावेज़ प्रामाणिक थे और दावा किए गए समय पर मौजूद थे)

प्रोटोकॉल की शब्दावली में यह किट किनसे रक्षा करती है:

- [FACT 05](facts/05_threat_model.txt) में T1–T3: दस्तावेज़ की सामग्री पर सार्वजनिक विवाद, यह दावा कि दस्तावेज़ दावा किए गए समय पर सीलबंद नहीं थे, पूर्व-तारीख से गढ़ने का दावा।
- T9: तृतीय पक्ष पत्रकारों को परिवर्तित प्रतियाँ सौंपते हैं।

यह किट किनसे रक्षा **नहीं** करती:

- व्हिसलब्लोअर के विरुद्ध व्यक्तिगत प्रतिशोध (T7)।
- व्हिसलब्लोअर पर वापसी (retraction) का दबाव (T4, आंशिक रूप से FACT 07 द्वारा कम किया गया जो मौन वापसी को सार्वजनिक रूप से पता-योग्य बनाता है)।
- स्वयं प्रकटीकरण के लिए व्हिसलब्लोअर के विरुद्ध कानूनी कार्रवाई (T5)।
- व्हिसलब्लोअर के चरित्र को बदनाम करना (T8)।
- मध्यस्थों द्वारा कहानी को चुपचाप छोड़ देना (T10)।

यदि आप व्हिसलब्लोअर हैं और इस किट का उपयोग करने पर विचार कर रहे हैं, तो **पहले इस पूरे README को पढ़ें**। प्रोटोकॉल एक उपकरण है। प्रोटोकॉल किसी वकील, पत्रकार, नियामक, या सावधानीपूर्ण परिचालन सुरक्षा (operational security) का स्थानापन्न नहीं है।

## यह किट क्या उत्पन्न करती है

साक्ष्य (साक्ष्य स्वयं नहीं) का एक क्रिप्टोग्राफिक रूप से सीलबंद *मेनिफेस्ट* जो सिद्ध करता है कि:

1. प्रत्येक साक्ष्य फ़ाइल के विशिष्ट बाइट सीलिंग के क्षण पर मौजूद थे।
2. सीलिंग किसी विशिष्ट समय पर या उससे पहले हुई (Bitcoin OTS प्रमाणन के माध्यम से)।
3. उद्गम (provenance), आरोप, खतरा मॉडल, प्राप्तकर्ता प्रोटोकॉल, और मौन-वापसी-नहीं प्रतिबद्धता — सभी एक साथ घोषित किए गए।

अंतर्निहित दस्तावेज़ तब तक आपके अधिकार में रहते हैं जब तक आप उन्हें किसी प्राप्तकर्ता को नहीं भेजते। प्राप्तकर्ता सत्यापित करते हैं कि उन्हें वही बाइट प्राप्त हुए जो आपने सील किए थे।

## इस किट का उपयोग कब करें

आपको पत्रकार, नियामक, या वकील को साक्ष्य भेजने **से पहले** मेनिफेस्ट को सील करने पर विचार करना चाहिए, यदि निम्न सभी सत्य हों:

- आपके पास डिजिटल रूप में साक्ष्य (PDF, ईमेल, छवियाँ, आदि) हैं जिन्हें आप बाइट-दर-बाइट प्रामाणिक रूप में संरक्षित करना चाहते हैं।
- आप अपेक्षा करते हैं कि मध्यस्थ साक्ष्य को बदल सकते हैं, विवादित कर सकते हैं, या दबा सकते हैं।
- आप संचरण तक अंतर्निहित दस्तावेज़ अपने अधिकार में रख सकते हैं।
- आप दस्तावेज़ों की सामग्री प्रकट किए **बिना** मेनिफेस्ट सार्वजनिक रूप से प्रकाशित कर सकते हैं (केवल हैश सार्वजनिक हैं)।
- आपने व्हिसलब्लोअर के रूप में सार्वजनिक रूप से पहचाने जाने के जोखिमों को तौला है (signing-principal क्षेत्र)। यदि गुमनाम हैं, तो सभी संबंधित प्रकाशनों में लगातार एक छद्म नाम (pseudonym) का प्रयोग करें।
- आपने अपने अधिकार-क्षेत्र में व्हिसलब्लोअर-संरक्षण कानून के बारे में किसी वकील से परामर्श किया है।

## इस किट का उपयोग कब **न** करें

इस किट का उपयोग न करें यदि:

- दस्तावेज़ों में स्वयं तृतीय पक्षों (रोगियों, ग्राहकों, कर्मचारियों) की व्यक्तिगत जानकारी है, जिसे सार्वजनिक रूप से हैश-योग्य नहीं बनाया जाना चाहिए। मूल नहीं, बल्कि एक संशोधित (redacted) संस्करण को हैश करें।
- मेनिफेस्ट प्रकाशित करने की क्रिया स्वयं व्हिसलब्लोअर को ऐसे ढंग से पहचान देगी जिसके लिए उन्होंने सहमति नहीं दी है।
- आपने अभी तक किसी वकील से परामर्श नहीं किया है। प्रोटोकॉल एक उपकरण है; व्हिसलब्लोअर-संरक्षण कानून अधिकार-क्षेत्र-विशिष्ट है और प्रोटोकॉल के बाहर रहता है।
- आप बाद में किसी एक भी आरोप को वापस लेना चाह सकते हैं। सील स्थायी है; नई सीलबंद सुधार-घोषणाएँ ही ईमानदारी से संशोधन का एकमात्र तरीका हैं।
- आप दबाव या समय के दबाव में यह कर रहे हैं। प्रोटोकॉल जबरदस्ती को बढ़ाता है (amplifies coercion); इनकार-द्वार (myriam-kit `SPEC §8`) प्रकाशकों की भाँति व्हिसलब्लोअरों पर भी लागू होता है।

## वास्तविक प्रकटीकरण के लिए इस किट को कैसे फ़ोर्क करें

```bash
# 1. इस उदाहरण को नई निर्देशिका में कॉपी करें
cp -r ~/Genesis/myriam-kit/examples/whistleblower-manifest ~/my-disclosure
cd ~/my-disclosure

# 2. काल्पनिक सामग्री हटाएँ; अपने वास्तविक तथ्यों से बदलें।
# facts/ के अंतर्गत प्रत्येक फ़ाइल एक तथ्य है। प्रत्येक को पढ़ें और बदलें।
$EDITOR facts/01_example_notice.txt   # वास्तविक कवर वक्तव्य में बदलें
$EDITOR facts/02_provenance.txt       # आपके साक्ष्य का वास्तविक उद्गम
$EDITOR facts/03_evidence_manifest.txt # अपने वास्तविक दस्तावेज़ों को हैश करें और सूचीबद्ध करें
$EDITOR facts/04_allegation_summary.txt # आपका वास्तविक आरोप
$EDITOR facts/05_threat_model.txt     # आपकी अपेक्षित वास्तविक प्रतिक्रियाएँ
$EDITOR facts/06_recipient_protocol.txt # अपने प्राप्तकर्ता के लिए समायोजित करें
$EDITOR facts/07_no_retraction_commitment.txt # आपकी वापसी-न-करने की घोषणा
$EDITOR facts/08_signing_principal.txt # आपकी वास्तविक हस्ताक्षरकर्ता पहचान (या छद्म नाम)

# 3. site/index.html को वास्तविक तथ्यों के अनुसार अद्यतन करें।
# seal.sh स्क्रिप्ट HTML को स्वतः नहीं भरती; यह मैन्युअल संपादन ही
# मानव-समीक्षा (human-review) चरण है।
$EDITOR site/index.html

# 4. सील करें।
bash build/seal.sh
# उत्पादन: merkle_root.txt, merkle_root.txt.ots, IPFS CID

# 5. स्थानीय रूप से सत्यापित करें।
python3 build/verify.py

# 6. मेनिफेस्ट (दस्तावेज़ नहीं) को इनमें से किसी पर प्रकाशित करें:
#    - आपके नियंत्रण का एक स्थैतिक होस्ट (Cloudflare Pages, GitHub Pages, आदि)
#    - मुद्रित CID पर IPFS नेटवर्क
#    - दोनों (अनुशंसित)

# 7. अलग से, अंतर्निहित दस्तावेज़ों को अपनी पसंद के किसी सुरक्षित
#    चैनल (Signal, SecureDrop, आदि) के माध्यम से प्राप्तकर्ता को भेजें।
#    प्राप्तकर्ता आपके मेनिफेस्ट के विरुद्ध फ़ाइल हैश सत्यापित करता है।
```

## वास्तविक साक्ष्य फ़ाइलों के लिए हैश की गणना

किसी दस्तावेज़ फ़ाइल को SHA-256 से हैश करने के लिए:

```bash
shasum -a 256 my_document.pdf
```

पहले 64 हेक्स अक्षर SHA-256 हैं। उसे मेनिफेस्ट में डालें।

आप एक साथ कई फ़ाइलों को हैश कर सकते हैं:

```bash
shasum -a 256 *.pdf *.eml *.jpg > MANIFEST.txt
```

इससे आपको `facts/03_evidence_manifest.txt` के लिए डेटा मिलता है।

## परिचालन सुरक्षा (opsec) संबंधी नोट्स

MYRIAM प्रोटोकॉल परिचालन सुरक्षा प्रदान **नहीं** करता है। यह *साक्ष्यात्मक स्थायित्व* (evidentiary durability) प्रदान करता है। आपको अभी भी इनकी आवश्यकता है:

- अपने वकील तक एक सुरक्षित संचार चैनल ([Signal](https://signal.org/), व्यक्तिगत रूप से, या आपके वकील का स्थापित सुरक्षित चैनल)
- अंतर्निहित दस्तावेज़ों के लिए एक सुरक्षित भंडारण स्थान (आराम-समय में एन्क्रिप्टेड, ऐसी क्लाउड सेवाओं पर नहीं जिन तक आपका नियोक्ता पहुँच सके)
- सीलिंग के लिए एक स्वच्छ डिवाइस (नियोक्ता खातों से न जुड़ा व्यक्तिगत लैपटॉप; आदर्शतः केवल इसी प्रयोजन से इस्तेमाल किया गया)
- प्रकाशन के लिए एक स्वच्छ नेटवर्क (नियोक्ता का नेटवर्क नहीं, न ही आपका घर का IP यदि नियोक्ता आपके ISP को समन कर सकता है)

व्हिसलब्लोअरों के लिए परिचालन सुरक्षा पर अनुशंसित पठन:

- [The Intercept का SecureDrop दस्तावेज़ीकरण](https://securedrop.org/)
- [Freedom of the Press Foundation के संसाधन](https://freedom.press/)
- [EFF की निगरानी आत्म-रक्षा मार्गदर्शिका](https://ssd.eff.org/)

सीलिंग प्रक्रिया शुरू करने से **पहले** उनसे परामर्श करें। प्रोटोकॉल अपने आह्वान से पहले लिए गए ख़राब opsec निर्णयों के विरुद्ध रक्षा नहीं करता।

## प्राप्तकर्ताओं को क्या जानना चाहिए

प्राप्तकर्ताओं को यह README सीलबंद मेनिफेस्ट के साथ भेजें। मेनिफेस्ट स्वयं प्राप्तकर्ता प्रोटोकॉल (FACT 06) समाहित करता है, परंतु यह README प्राप्तकर्ता की भूमिका का विस्तार करता है:

- **पत्रकार**: सबसे पहले स्वयं मेनिफेस्ट की अखंडता सत्यापित करें। फिर एक अलग सुरक्षित चैनल के माध्यम से दस्तावेज़ों का अनुरोध करें। प्रकाशन से पहले प्रत्येक दस्तावेज़ का SHA-256 मेनिफेस्ट के विरुद्ध मेल खाता है, यह सत्यापित करें। व्हिसलब्लोअर की सहमति के बिना दस्तावेज़ प्रकाशित **न** करें; सीलबंद मेनिफेस्ट के अस्तित्व और अपने सत्यापन निष्कर्ष को प्रकाशित करें।
- **नियामक**: वही सत्यापन प्रक्रिया। सीलबंद मेनिफेस्ट + मेल खाते दस्तावेज़ औपचारिक कस्टडी-श्रृंखला संग्रह के बाहर उपलब्ध सबसे मज़बूत साक्ष्य रूप हैं।
- **वकील**: सीलबंद मेनिफेस्ट दावा किए गए समय पर दस्तावेज़ों के अस्तित्व के लिए फॉरेंसिक एंकरिंग प्रदान करता है। क्या यह किसी विशिष्ट अधिकार-क्षेत्र के साक्ष्य-नियमों को संतुष्ट करता है, यह आपके निर्धारण के लिए है।

## यह किट क्या **नहीं** है

- कानूनी सलाह का स्थानापन्न नहीं।
- कोई गारंटी नहीं कि आपके प्रकटीकरण पर कोई कार्रवाई करेगा।
- प्रतिशोध के विरुद्ध कोई संरक्षण नहीं।
- दस्तावेज़ों की सामग्री का प्रमाणीकरण नहीं (केवल सीलिंग-के-समय के बाइट्स का)।
- किसी से गुमनाम रूप से संवाद करने का कोई तरीका नहीं — सील सार्वजनिक है; यदि आपका हस्ताक्षरकर्ता-नाम वास्तविक है, तो आप पहचाने जाते हैं।

## समापन

एक सीलबंद मेनिफेस्ट एक छोटी चीज़ है। यह एक बाइट-स्ट्रीम है जो सिद्ध करती है कि बाइट-स्ट्रीमों का एक बड़ा समूह किसी विशिष्ट समय पर किसी विशिष्ट रूप में मौजूद था। उसी छोटेपन में उसकी ताक़त है: वह केवल एक काम करती है, और वह एक काम Bitcoin ब्लॉकचेन के पूरे जीवनकाल तक करती है।

यदि आप यह इसलिए पढ़ रहे हैं क्योंकि आप व्हिसलब्लोअर बनने पर विचार कर रहे हैं, तो कृपया: पहले किसी वकील से बात करें। ऊपर दी गई परिचालन सुरक्षा मार्गदर्शिकाओं का उपयोग करें। फिर, यदि और केवल यदि वे सभी उच्चस्तरीय विचार क्रम में हैं, तो इस किट को अपने प्रकटीकरण की क्रिप्टोग्राफिक रीढ़ के रूप में विचार करें। क्रिप्टो सरल हिस्सा है। क्रिप्टो से ऊपर का सब कुछ — वही वह जगह है जहाँ वास्तविक जोखिम रहता है।
