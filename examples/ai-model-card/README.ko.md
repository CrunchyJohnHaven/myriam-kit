# AI 모델 거동 잠금 키트(Behavior Lock Kit)

*한국어 번역 — 원본 영어 버전은 [README.md](README.md) 참조*

모델을 출시하면서, 출시의 주장 — 가중치(weights) 동일성, 기능,
평가 점수, 안전성 검토, 배포 전 약속(commitments) — 을
**암호학적으로 앵커링되고 공개적으로 검증 가능하게** 만들고자 하는
AI 연구소(상업 또는 오픈소스)를 위한 턴키 myriam-kit 예제입니다.

본 키트는 배포된 모델이 조용히 표류하거나, 평가 점수가 사후에 고쳐지거나,
배포 전 주장이 수년 후에 "우리는 그런 말을 한 적이 없다"라는 식으로
다투어지는 특정 실패 양식에 대해 방어합니다.

## 본 키트가 산출하는 것

하나의 모델 출시에 대해 다음을 담고 있는, 암호학적으로 봉인된 매니페스트:

1. 모델 식별 정보(이름, 아키텍처, 파라미터 수, 라이선스)
2. **가중치 파일의 SHA-256** — 정본 아티팩트 해시
3. 모델 카드 텍스트(기능, 의도된 사용, 범위 외 사용, 알려진 실패 양식)
4. 출시 시점의 평가 결과(기능 평가, 안전성 평가, 캘리브레이션)
5. 배포 전 약속(출시 후 조용히 변경하지 않겠다고 약속하는 사항들)
6. 서명 책임자(책임을 지는 인간 + 조직)

봉인은 각각의 항목을 바이트 단위로 안정화하고, OpenTimestamps를 통해
Bitcoin에 시간적으로 앵커링하며, 어떠한 단일 호스팅 주체에도 의존하지 않고
IPFS를 통해 회수 가능하게 만듭니다.

작동하는 가상의 예시: [facts/](facts/) 및 렌더된
[site/index.html](site/index.html)을 참조하십시오. 본 예제는 가상의 모델
("Calm-mini-v0")을 사용합니다. 실제 출시에서는 예시 내용을 대체하게 됩니다.

## 본 키트가 방어하는 것

**봉인이 직접 포착하는 실패 양식:**

- **가중치 조작** — 출시된 가중치 파일의 SHA-256이 FACT 03에 들어 있습니다.
  사용자가 다운로드하는 바이트에 대한 출시 후의 어떠한 변경도 재해싱으로
  검출 가능합니다.
- **조용한 평가 결과 사후 수정(retcon)** — 출시 시점의 평가 점수는
  FACT 05에 들어 있습니다. 향후 "평가 점수는 항상 X였다"라는 분쟁에
  공개적인 앵커가 존재하게 됩니다.
- **기능 주장(claim)의 표류** — FACT 04의 모델 카드 텍스트는 바이트
  단위로 안정적입니다. 향후 "우리는 기능 X를 주장한 적이 없다"라는
  분쟁은 봉인된 텍스트에 대해 공개적으로 검증 가능합니다.
- **조용한 파인튜닝 재배포** — 작동 예시의 FACT 06 배포 전 약속 #1은
  "동일한 이름 아래 비밀스러운 파인튜닝 금지"입니다. 검출 방식: 새로운
  가중치 해시가 등장한다는 것은 곧 새 모델명과 새로운 봉인된 공표를
  의미하며, 그렇지 않다면 약속이 깨진 것입니다.
- **숨겨진 실패 양식** — FACT 04는 알려진 실패 양식 목록에 약속을
  답니다. 이 목록은 단조적(monotonic)입니다 — 동결된 체크포인트에서는
  증가할 수 있을 뿐 축소될 수 없습니다. 연구소가 내부적으로는 알고
  있었지만 공표하지 않은 실패 양식이 향후 발견되면, 이는 다툴 수 있는
  증거가 됩니다.

**커버되지 *않는* 실패 양식:**

- 다운스트림 사용자에 의한 모델 오용(봉인은 무엇이 출시되었는지를
  증명할 뿐, 사용자가 그것으로 무엇을 하는지는 증명하지 않습니다).
- 제3자에 의한 적대적 파인튜닝(제3자 파인튜닝은 다른 가중치 해시를
  산출하므로 원래 봉인은 영향을 받지 않지만, 제3자의 모델은 봉인의
  보호 범위 밖에 있습니다).
- 배포 환경 변경(다른 시스템 프롬프트, RAG 구성 등)에 따른 거동 변화.
- 출시 시점에 연구소가 실로 알지 못했던 실패. 봉인은 연구소가 *알고
  있던 것*에 대해 약속하게 하는 것이지, 알지 못했던 것을 앵커링할
  수는 없습니다.

## 본 키트를 사용해야 할 때

다음의 경우 모델 카드를 봉인할 것을 고려해야 합니다.

- 모델 가중치를 공개적으로 출시하거나, 또는 다운스트림 사용자가 거동
  안정성에 관심을 갖는 맥락에서 모델을 배포할 때.
- 향후 "우리는 그런 말을 한 적이 없다"라는 분쟁이 그럴듯할 만큼
  출시가 충분히 중요한 경우(대규모로 배포되는 모든 모델, 안전이 중요한
  모든 모델, 기능 주장이 동반되는 모든 출시).
- 귀 조직이 향후 행동을 제약하는 배포 전 약속을 기꺼이 할 의향이 있는
  경우(이것이 핵심 — 다음 절 참조).
- 출시 시점에 가중치 파일의 SHA-256을 계산할 수 있는 경우.

## 본 키트를 사용해서는 안 될 때

다음의 경우 본 키트를 사용하지 **마십시오**.

- 예시 FACT 06의 배포 전 약속에 구속되기를 원치 않는 경우. 봉인은
  약속을 기계적으로 검출 가능하게 만듭니다. 그러한 책임을 감당할
  준비가 되어 있지 않다면 봉인하지 마십시오.
- 가중치 파일이 자주 변경될 예정인 경우(예: 지속적 파인튜닝
  파이프라인). 가중치가 변경될 때마다 새로운 봉인 출시가 필요합니다.
  출시마다 봉인하는 것은 무거운 절차로, 본 키트는 *동결된*
  체크포인트를 위한 것입니다.
- 안전성 검토를 아직 수행하지 않은 경우. 봉인은 안전성 검토의 결과를
  약속으로 만듭니다. 불완전한 검토를 봉인한다면 불완전한 검토에 대한
  영구적인 공개 기록이 남게 됩니다.
- 추후에 특정 평가 점수를 철회하고 싶을 가능성이 있는 경우. 봉인은
  영구적입니다. 새로운 봉인 공표가 대체할 수는 있어도, 원본 자체는
  되돌릴 수 없습니다.

## 핵심을 이루는 배포 전 약속

예시의 FACT 06은 공표자에게 다음을 약속하게 합니다.

1. **동일한 이름 아래 비밀스러운 파인튜닝 금지.** 어떠한 파인튜닝도
   새로운 모델명, 새로운 봉인 공표, 새로운 가중치 해시를 산출합니다.
   선행 모델은 정본적으로 그 이름의 버전으로 남습니다.

2. **비밀스러운 평가 갱신 금지.** 재평가는 원본을 Merkle 루트로 참조하는
   새로운 봉인 공표로 이루어집니다. 버전은 추적되며, 덮어쓰이지 않습니다.

3. **알려진 실패 양식은 늘어날 수 있을 뿐, 결코 줄어들 수 없다.** 동결된
   체크포인트에서는 실패 양식이 고쳐질 수 없습니다. 따라서 공표된 목록은
   사실상 단조적입니다.

4. **테이크다운(takedown) 처리.** 배포에서 모델을 제거하도록 강제될 경우,
   연구소는 그 테이크다운에 관한 최종 봉인 진술을 공표할 것을 약속합니다.

본 키트를 채택하는 연구소는 이 약속들 중 어느 것을 실제로 이행할 수
있는지를 결정해야 합니다. 네 가지 모두를 이행할 수 없다면, 봉인 *이전*에
이행할 수 있는 것을 반영하도록 FACT 06을 수정하고, 그 결과 약화되는
보호 수준을 받아들이십시오. 이행할 의사가 없는 약속을 봉인하는 것은
아예 봉인하지 않는 것보다도 더 나쁩니다.

## 실제 출시를 위해 본 키트를 포크하는 방법

```bash
# 1. 이 예제를 출시 파이프라인으로 복사합니다
cp -r ~/Genesis/myriam-kit/examples/ai-model-card ~/my-model-release
cd ~/my-model-release

# 2. 실제 출시에 맞게 각 fact을 편집합니다
$EDITOR facts/01_example_notice.txt   # 제거 — 귀하의 출시는 예시가 아닙니다
                                       # 귀하의 모델에 대한 출시 공지로 교체하십시오
$EDITOR facts/02_model_identification.txt
$EDITOR facts/03_weights_hash.txt     # shasum -a 256 your_weights.safetensors
$EDITOR facts/04_model_card.txt       # 정본 모델 카드
$EDITOR facts/05_eval_results.txt     # 실제 평가 결과
$EDITOR facts/06_release_commitments.txt # 귀하가 이행할 수 있는 약속들

# 3. site/index.html을 실제 출시에 맞게 갱신합니다.

# 4. 봉인합니다.
bash build/seal.sh

# 5. 로컬에서 검증합니다.
python3 build/verify.py

# 6. 다음 위치에 배포합니다:
#    - 귀하의 출시 페이지(모델 카드 및 가중치 옆)
#    - IPFS 네트워크(seal.sh 스크립트가 이미 수행함)
#    - 선택 사항: Hugging Face / GitHub 출시 노트에서 봉인 공표로 링크
```

## 기존 출시 파이프라인과의 통합

본 키트는 기존 도구를 대체하는 것이 아니라, 함께 구성되도록 설계되었습니다.

- **Hugging Face**: 모델 카드의 README.md에 봉인 공표의 URL이나 IPFS CID를
  포함시키십시오. 바이트 단위로 안정적인 출시 주장을 위해 독자를 봉인으로
  안내하십시오.
- **GitHub releases**: OpenTimestamps 영수증(`merkle_root.txt.ots`)을 출시
  자산(release asset)으로 포함시키십시오. "출시가 변경되었다"라는 향후의
  분쟁이 Bitcoin에 앵커링됩니다.
- **모델 카드 표준 (Mitchell et al. 2019)**: 본 키트는 호환됩니다. 표준
  모델 카드의 필드는 봉인의 FACTS 02–05에 대응됩니다. 평소대로 모델
  카드를 작성하고, 추가로 그것을 봉인하십시오.
- **평가 하니스 (lm-evaluation-harness, BIG-bench 등)**: 정확한 하니스
  커밋과 원시 출력 JSON을 해시한 뒤, 그 해시들을 FACT 05에서 참조하십시오.
  그러면 향후의 재실행이 독립적으로 재현 가능해집니다.

## 가중치 해시 계산

단일 safetensors 파일의 경우:

```bash
shasum -a 256 model.safetensors
```

다중 파일 체크포인트(GGUF 샤드, 샤딩된 safetensors)의 경우:

```bash
# 해시들을 정렬된 순서로 이어 붙인 뒤 다시 해시합니다
for f in $(ls model-*.safetensors | sort); do
  shasum -a 256 "$f"
done > weights_manifest.txt
shasum -a 256 weights_manifest.txt
```

최종 해시가 FACT 03에 들어가는 값입니다. 사용자가 다운로드하는 동안 개별
샤드를 검증할 수 있도록, 샤드별 해시와 최종 집계 해시를 모두 포함시키십시오.

## 수신자(귀하 모델의 사용자)가 알아야 할 것

MYRIAM 봉인 공표를 동반한 모델 출시는 사용자에게 다음을 허용합니다.

1. **다운로드한 가중치 검증** — 재해싱하여 FACT 03과 비교.
2. **모델 카드가 사후 수정되지 않았음을 검증** — 봉인된 FACT 04를 연구소의
   현재 공표 버전과 비교.
3. **평가 점수가 출시 시점의 것임을 검증** — 연구소의 현재 평가 주장을
   봉인된 FACT 05와 비교.
4. **연구소의 배포 전 약속에 대한 책임 추궁** — 실제 거동을 봉인된 FACT 06과
   비교.

불일치를 발견한 사용자는 그 불일치에 대한 암호학적 증거를 보유합니다.
사용자는 그 증거를 공표할 수 있으며, 원본 봉인 공표가 그 앵커가 됩니다.

## 본 키트가 **아닌** 것

- 실제 안전성 검토를 수행하는 일의 대체물이 아닙니다. 봉인은 안전성
  검토의 *기록*이지, 검토 그 자체가 아닙니다.
- 실제 평가의 대체물이 아닙니다. 봉인은 귀하가 산출한 점수를 앵커링할
  뿐, 방법론을 검증하지 않습니다.
- 모델이 안전하다는 보장이 아닙니다. 봉인은 연구소가 출시 시점에 모델에
  대해 *주장하는 바*를 기록할 뿐, 그러한 주장을 인증하지 않습니다.
- 다운스트림 사용자에 의한 모델 오용에 대한 보호가 아닙니다.

## 적대적 파인튜닝에 관한 비고

흔한 AI 안전성 우려: 누군가 귀하의 모델을 안전 거동을 제거하도록
파인튜닝한 뒤 재배포하는 경우. 봉인은 이를 막을 수 **없습니다**.
봉인이 할 수 있는 것은 다음과 같습니다.

- *원래* 모델이 무엇을 해야 했는지를 앵커링(FACT 04).
- 원래 평가 점수가 무엇이었는지를 앵커링(FACT 05).
- 적대적 파인튜닝을 측정할 수 있는 기준점을 제공.

제3자 재배포에 마주친 사용자는 가중치가 원본의 봉인 해시와 일치하는지를
검증할 수 있습니다. 일치하지 않는다면, 사용자는 비원본 모델을 갖고 있음을
알고, 연구소에 귀속된 어떠한 안전성 주장도 적용되지 않음을 알게 됩니다.

## 맺음말

모델 출시를 봉인하는 일은 연구소에게는 소규모의 추가 작업입니다. 해시
계산과 봉인 스크립트 실행에 ~10분, 어떤 배포 전 약속을 포함할지에 대한
30초의 결정. 봉인이 더하는 책임은 비대칭적입니다. 정직한 연구소에는 작은
