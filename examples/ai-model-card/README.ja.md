# AI モデル挙動ロックキット(AI Model Behavior Lock Kit)

*日本語訳 — 英語の原文は [README.md](README.md) を参照*

商用・オープンソースを問わず、モデルをリリースする AI 研究所(AI lab)が、リリースに伴う主張 — 重み(weights)の同一性、能力(capabilities)、評価スコア、安全性レビュー、デプロイ前のコミットメント — を **暗号学的に固定し、公的に検証可能にする** ためのターンキー myriam-kit サンプルです。

本キットは、デプロイされたモデルが密かにドリフトしたり、評価スコアが事後的に書き換えられたり、デプロイ前の主張が数年後に「そんなことは言っていない」と争われる、という特定の失敗モードに対する防御を提供します。

## 本キットが生成するもの

1 つのモデルリリースについて、暗号学的に封緘された以下のマニフェスト:

1. モデル識別情報(名称、アーキテクチャ、パラメータ数、ライセンス)
2. **重みファイルの SHA-256** — 正式な成果物ハッシュ
3. モデルカード本文(能力、想定用途、対象外用途、既知の失敗モード)
4. リリース時の評価結果(能力評価、安全性評価、較正(calibration))
5. デプロイ前のコミットメント(リリース後に密かに変更しないことを約束する事項)
6. 署名主体(責任を負う人物 + 組織)

封緘により、これらの各項目はバイト単位で安定し、OpenTimestamps を介して Bitcoin に時刻固定され、いかなる単一のホスティング当事者にも依存せずに IPFS 経由で取得可能になります。

仮想例の作業済みサンプル: [facts/](facts/) およびレンダリング済みの [site/index.html](site/index.html) を参照してください。サンプルでは架空のモデル("Calm-mini-v0")を使用しています。実際のリリースではサンプル内容を置き換えてください。

## 本キットが防御する失敗モード

**封緘によって直接捕捉される失敗モード:**

- **重みの改ざん(Weight tampering)** — リリースされた重みファイルの SHA-256 は FACT 03 に含まれます。ユーザーがダウンロードするバイト列のリリース後の改変は、再ハッシュにより検出可能です。
- **静かな評価の事後改変(Silent eval retcons)** — リリース時点の評価スコアは FACT 05 にあります。「評価スコアは元から X だった」と将来主張される場合の公的なアンカーとなります。
- **能力主張のドリフト(Capability claim drift)** — FACT 04 のモデルカード本文はバイト単位で安定します。「能力 X を主張したことはない」という将来の主張は、封緘された本文に対して公的に照合可能です。
- **静かなファインチューン再配布(Silent fine-tune redistribution)** — 作業済みサンプルにおける FACT 06 のデプロイ前コミットメント #1 は「同一名でのサイレントファインチューニング禁止」です。検出: 新しい重みハッシュは新しいモデル名と新しい封緘公開を意味します。さもなくば、コミットメントが破られたことになります。
- **隠された失敗モード(Hidden failure modes)** — FACT 04 は既知失敗モードのリストにコミットします。リストは単調(monotonic)です(凍結されたチェックポイントについては増えることはあっても減ることはない)。研究所が内部で知っていたが公表しなかった失敗モードが将来発覚した場合、争点となる証拠になります。

**カバーされない失敗モード:**

- 下流のユーザーによるモデルの誤用(封緘はリリースされたものを証明するのであって、ユーザーがそれをどう使うかを証明するものではありません)。
- 第三者による敵対的ファインチューニング(第三者のファインチューンは異なる重みハッシュを生むため、オリジナルの封緘には影響しませんが、その第三者のモデルは封緘の保護範囲外となります)。
- デプロイ環境の変化(異なるシステムプロンプト、RAG 構成等)に起因する挙動の変化。
- リリース時点で研究所が本当に知らなかった失敗。封緘は研究所が **知っていたこと** をコミットさせるのであり、知らなかったことを固定することはできません。

## 本キットを使うべきとき

以下に該当する場合、モデルカードの封緘を検討すべきです。

- モデル重みを公開リリースする、または下流ユーザーが挙動の安定性を気にする文脈でモデルをデプロイする場合。
- リリースが、将来「そんなことは言っていない」という争いが起こりうるほどに重要である場合(大規模デプロイされるあらゆるモデル、安全性が重要なあらゆるモデル、能力に関する主張を伴うあらゆるリリース)。
- 組織として、将来の行動を制約するデプロイ前コミットメントを行う意思がある場合(これは要(かなめ)です — 次節を参照)。
- リリース時に重みファイルの SHA-256 を計算可能である場合。

## 本キットを使うべきでないとき

以下に該当する場合は **使用しないでください**。

- サンプルの FACT 06 にあるデプロイ前コミットメントに拘束されたくない場合。封緘はコミットメントを機械的に検出可能にします。その説明責任に備える準備がないなら、封緘してはいけません。
- 重みファイルが頻繁に変わる場合(例: 連続的なファインチューニングパイプライン)。重みの変更ごとに新たな封緘リリースが必要です。リリース毎の封緘は重い処理であり、本キットは **凍結された** チェックポイント向けです。
- 安全性レビューがまだ完了していない場合。封緘は安全性レビューの所見をコミットします。不完全なレビューを封緘すると、不完全なレビューの恒久的な公的記録が残ることになります。
- 特定の評価スコアを後で撤回したい可能性がある場合。封緘は永続的です。新しい封緘公開は上書き(supersede)できますが、元のものを取り消すことはできません。

## 要(かなめ)となるデプロイ前コミットメント

サンプルの FACT 06 は発行者に以下をコミットさせます。

1. **同一名でのサイレントファインチューニング禁止。** いかなるファインチューンも新しいモデル名、新しい封緘公開、新しい重みハッシュを生みます。先行版は正準的にその名前のバージョンであり続けます。

2. **サイレントな評価更新禁止。** 再評価は、元のものを Merkle ルートで参照する新しい封緘公開として行います。バージョンは上書きされず、追跡されます。

3. **既知の失敗モードは増えることはあっても減ることはない。** 凍結されたチェックポイントの失敗モードは修正できません。したがって公表リストは事実上単調です。

4. **テイクダウン対応。** 配布からモデルを除去するよう強制された場合、研究所はテイクダウンに関する最終的な封緘声明を公表することにコミットします。

本キットを採用する研究所は、これらのうちどれを実際に履行できるかを決定する必要があります。4 つすべてを履行できないのであれば、封緘 **前** に FACT 06 を、履行できる内容を反映するように修正し、結果として保護が弱まることを受け入れてください。履行する意図のないコミットメントを封緘することは、封緘しないことよりも悪い結果になります。

## 実際のリリース向けに本キットをフォークする方法

```bash
# 1. 本サンプルをリリースパイプラインにコピーする
cp -r ~/Genesis/myriam-kit/examples/ai-model-card ~/my-model-release
cd ~/my-model-release

# 2. 実際のリリース向けに各 fact を編集する
$EDITOR facts/01_example_notice.txt   # 削除 — あなたのリリースはサンプルではない
                                       # あなたのモデル向けのリリース告知に置き換える
$EDITOR facts/02_model_identification.txt
$EDITOR facts/03_weights_hash.txt     # shasum -a 256 your_weights.safetensors
$EDITOR facts/04_model_card.txt       # 正準的なモデルカード
$EDITOR facts/05_eval_results.txt     # 実際の評価結果
$EDITOR facts/06_release_commitments.txt # 履行可能なコミットメント

# 3. site/index.html を実際のリリースを反映するよう更新する。

# 4. 封緘する。
bash build/seal.sh

# 5. ローカルで検証する。
python3 build/verify.py

# 6. デプロイ先:
#    - 自分のリリースページ(モデルカードと重みの隣)
#    - IPFS ネットワーク(seal.sh スクリプトが既に実施)
#    - 任意: Hugging Face / GitHub のリリースノートから封緘公開へリンクする
```

## 既存リリースパイプラインとの統合

本キットは既存ツールを置き換えるのではなく、それらと組み合わせて機能するように設計されています。

- **Hugging Face**: 封緘公開の URL または IPFS CID を、モデルカードの README.md に含めてください。バイト単位で安定したリリース主張のために、読者を封緘へ誘導します。
- **GitHub リリース**: OpenTimestamps レシート(`merkle_root.txt.ots`)をリリース資産として含めてください。「リリースが変更された」という将来の争いは、これにより Bitcoin に固定されます。
- **モデルカード標準(Mitchell et al. 2019)**: 本キットは互換性があります。標準のモデルカードフィールドは封緘の FACTS 02–05 にマップされます。通常どおりモデルカードを作成し、加えて封緘してください。
- **評価ハーネス(lm-evaluation-harness、BIG-bench 等)**: ハーネスの正確なコミットと生の出力 JSON をハッシュし、それらのハッシュを FACT 05 で参照してください。将来の再実行は独立に再現可能になります。

## 重みハッシュの計算

単一の safetensors ファイルの場合:

```bash
shasum -a 256 model.safetensors
```

複数ファイルのチェックポイント(GGUF シャード、シャード化された safetensors)の場合:

```bash
# ソート順にハッシュを連結し、再度ハッシュする
for f in $(ls model-*.safetensors | sort); do
  shasum -a 256 "$f"
done > weights_manifest.txt
shasum -a 256 weights_manifest.txt
```

最終ハッシュが FACT 03 に入るものです。ユーザーが個々のシャードをダウンロード時に検証できるよう、シャード単位のハッシュと最終的な集約ハッシュの両方を含めてください。

## 受信者(あなたのモデルのユーザー)が知っておくべきこと

MYRIAM 封緘公開を伴うモデルリリースは、ユーザーに以下を可能にします。

1. **ダウンロードした重みの検証** — 再ハッシュして FACT 03 と比較する。
2. **モデルカードが事後改変されていないことの検証** — 封緘された FACT 04 を、研究所の現在の公表版と照合する。
3. **評価スコアがリリース時のものであることの検証** — 研究所の現在の評価主張を、封緘された FACT 05 と比較する。
4. **研究所のデプロイ前コミットメントに対する説明責任の追及** — 実際の挙動を、封緘された FACT 06 と照合する。

不一致を発見したユーザーは、その不一致の暗号学的証拠を有することになります。ユーザーはその証拠を公表することができ、元の封緘公開がそのアンカーとなります。

## 本キットが **そうでない** もの

- 安全性レビューを実際に実施することの代替ではありません。封緘は安全性レビューの **記録** であって、レビュー自体ではありません。
- 実際の評価の代替ではありません。封緘はあなたが算出したスコアを固定するのであり、方法論を検証するものではありません。
- モデルが安全であることの保証ではありません。封緘はリリース時点で研究所がモデルについて主張する内容を文書化するのであって、それらの主張を認証するものではありません。
- 下流ユーザーによるモデルの誤用に対する保護ではありません。

## 敵対的ファインチューニングについての注記

一般的な AI 安全性の懸念事項として、あなたのモデルを誰かが安全挙動を除去するようファインチューンして再配布する、というものがあります。封緘はこれを防止 **できません**。封緘ができるのは以下です。

- **元の** モデルが何をするはずだったかを固定する(FACT 04)。
- 元の評価スコアが何であったかを固定する(FACT 05)。
- 敵対的ファインチューンを測るための基準点を提供する。

第三者の再配布物に遭遇したユーザーは、重みが元の封緘ハッシュと一致するかを検証できます。一致しない場合、ユーザーは非オリジナルのモデルを扱っていること、および研究所に帰属するいかなる安全性主張も適用されないことを知ることができます。

## おわりに

モデルリリースの封緘は研究所にとって小さな追加作業です: ハッシュの計算と封緘スクリプトの実行に約 10 分、そしてどのデプロイ前コミットメントを含めるかについての 30 秒の判断です。それがもたらす説明責任は非対称です: 誠実な研究所にとってはコストが小さく、主張を密かにドリフトさせようとする研究所にとっては大きな制約となります。

採用を検討する AI 研究所の方へ: 本キットはパブリックドメイン(CC0)です。フォークしてください。修正してください。要(かなめ)は数学であって、特定の実装ではありません。
