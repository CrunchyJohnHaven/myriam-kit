# AI 模型行为锁定套件

一个面向 AI 实验室(商业或开源)的开箱即用 myriam-kit 范例,适用于
发布模型并希望让发布所作出的各项声明 —— 权重身份、能力范围、评测分数、
安全审查、部署前承诺 —— **在密码学上得到锚定且可公开验证**的场景。

本套件防御的具体失败模式是:已部署的模型悄然漂移、评测分数被事后追改,
或部署前的声明在多年后被以"我们从未那样说过"为由提出争议。

## 本套件产生什么

一份在密码学上密封的清单,针对一次模型发布,包含:

1. 模型身份信息(名称、架构、参数数量、许可证)
2. **权重文件的 SHA-256** —— 权重产物的规范哈希
3. 模型卡文本(能力范围、预期用途、超出范围的用途、已知失败模式)
4. 发布时的评测结果(能力评测、安全评测、校准)
5. 部署前承诺(您承诺在发布之后不会悄然更改的事项)
6. 签署主体(承担责任的人 + 组织)

密封使上述每一项都达到字节级稳定,通过 OpenTimestamps 锚定到 Bitcoin
的时间线,并可通过 IPFS 取回,独立于任何单一托管方。

请参阅完整的虚构示例:[facts/](facts/) 与渲染后的
[site/index.html](site/index.html)。该示例使用一个虚构模型
("Calm-mini-v0");真实发布会用真实内容替换示例内容。

## 本套件防御什么

**密封直接捕捉的失败模式:**

- **权重篡改** —— 已发布权重文件的 SHA-256 写在事实 03 中。用户下载
  到的字节在发布后若有任何改动,均可通过重新哈希被检测出来。
- **评测分数被悄然追改** —— 发布时的评测分数写在事实 05 中。未来
  "评测分数一直是 X"一类的争议有了公开锚点。
- **能力声明的漂移** —— 事实 04 中的模型卡文本字节级稳定。未来
  "我们从未声称具备能力 X"一类的争议,可公开对照已密封的文本进行核查。
- **悄然进行的微调再分发** —— 事实 06 中的部署前承诺 1(在该完整示例
  中)是"不在同一名称下进行悄然的微调"。检测方式:任何新的权重哈希
  都意味着一个新的模型名称和一次新的密封发布,否则承诺即被违反。
- **隐瞒的失败模式** —— 事实 04 承诺一份已知失败模式清单。对于一个
  冻结的检查点,该清单是单调的(只能增长,不能缩减)。日后被发现的
  失败模式,若实验室内部早已知晓但未公布,即可成为可争议的证据。

**不在覆盖范围内的失败模式:**

- 模型被下游用户滥用(密封证明发布出来的是什么,而非用户拿它做什么)。
- 第三方进行的对抗性微调(第三方微调会产生不同的权重哈希,因此原始
  密封不受影响,但第三方的模型不在密封的保护范围之内)。
- 因部署环境变化引起的行为变化(不同的系统提示词、不同的 RAG 配置
  等)。
- 实验室在发布时确实不知道的失败。密封将实验室约束在 *他们已知的范围*
  之内;它无法锚定他们尚未知道的内容。

## 何时使用本套件

如有以下情形,您应考虑密封一份模型卡:

- 您正在公开发布模型权重,或在下游用户关心行为稳定性的场景中部署
  模型。
- 该发布足够重要,以至于未来出现"我们从未那样说过"一类的争议是合理
  可能的(任何大规模部署的模型;任何安全关键模型;任何附带能力声明的
  发布)。
- 您的组织愿意作出会约束未来行动的部署前承诺(这是承重的一点 ——
  见下一节)。
- 您能在发布时刻计算权重文件的 SHA-256。

## 何时**不要**使用本套件

如有以下情形,请**不要**使用本套件:

- 您不愿被示例事实 06 中的部署前承诺所约束。密封使承诺变得可机械
  检测;若您尚未准备好承担这种问责,就不要密封。
- 权重文件会频繁变更(例如持续微调流水线)。每次权重变更都需要一次
  新的密封发布。按发布密封负担较重;本套件面向 *冻结的* 检查点。
- 安全审查尚未完成。密封会承诺安全审查的结论。若您对一份尚不完整的
  审查进行密封,就会得到一份不完整审查的永久公开记录。
- 您日后可能想撤回任何具体的评测分数。密封是永久的。新的密封发布
  可以取代旧版,但原始版本无法被消除。

## 承重的部署前承诺

示例中的事实 06 将发布方约束于:

1. **不在同一名称下进行悄然的微调。** 任何微调都会产生一个新的模型
   名称、一次新的密封发布以及一个新的权重哈希。前身在规范意义上仍是
   原名称对应的版本。

2. **不进行悄然的评测更新。** 重新评测会产生新的密封发布,通过 Merkle
   根引用原始版本。版本被追踪,而非被覆盖。

3. **已知失败模式只能增加,绝不缩减。** 一个冻结的检查点无法修复其
   失败模式。因此已发布的清单在效力上是单调的。

4. **下架处置。** 若被迫将模型从分发渠道中移除,实验室承诺发布一份
   关于此次下架的最终密封声明。

采用本套件的实验室需要决定:上述哪些承诺它确实能够履行。如果您无法
履行全部四项,请在密封 *之前* 修改事实 06,以反映您确实能履行的内容,
并接受由此减弱的保护强度。密封自己并不打算履行的承诺,比根本不密封
还要糟糕。

## 如何为真实发布派生本套件

```bash
# 1. 将本示例复制进您的发布流水线
cp -r ~/Genesis/myriam-kit/examples/ai-model-card ~/my-model-release
cd ~/my-model-release

# 2. 为您的真实发布编辑每一项事实
$EDITOR facts/01_example_notice.txt   # 删除 —— 您的发布不是示例
                                       # 替换为您模型的发布说明
$EDITOR facts/02_model_identification.txt
$EDITOR facts/03_weights_hash.txt     # shasum -a 256 your_weights.safetensors
$EDITOR facts/04_model_card.txt       # 规范模型卡
$EDITOR facts/05_eval_results.txt     # 真实的评测结果
$EDITOR facts/06_release_commitments.txt # 您能履行哪些承诺

# 3. 更新 site/index.html,使其反映真实发布。

# 4. 密封。
bash build/seal.sh

# 5. 本地验证。
python3 build/verify.py

# 6. 部署至:
#    - 您的发布页面(放在模型卡与权重旁边)
#    - IPFS 网络(seal.sh 脚本已经完成了这一步)
#    - 可选:在您的 Hugging Face / GitHub 发布说明中链接至已密封的发布
```

## 与现有发布流水线的集成

本套件的设计目标是与现有工具组合,而非替代它们:

- **Hugging Face**:在模型卡的 README.md 中包含已密封发布的 URL 或
  IPFS CID。指引读者前往密封,获取字节级稳定的发布声明。
- **GitHub releases**:将 OpenTimestamps 收据(`merkle_root.txt.ots`)
  作为 release 资产附上。未来"发布被改动过"一类的争议即被锚定到
  Bitcoin。
- **模型卡标准(Mitchell et al. 2019)**:本套件与该标准兼容。标准
  模型卡字段映射到密封中的事实 02–05。您照常产出模型卡,然后额外
  对其密封。
- **评测框架(lm-evaluation-harness、BIG-bench 等)**:对所用框架的
  具体 commit 与原始输出 JSON 进行哈希;在事实 05 中引用这些哈希。
  日后的复跑因此可被独立复现。

## 计算权重哈希

对于单一 safetensors 文件:

```bash
shasum -a 256 model.safetensors
```

对于多文件检查点(GGUF 分片、分片的 safetensors):

```bash
# 按排序顺序拼接哈希,再次进行哈希
for f in $(ls model-*.safetensors | sort); do
  shasum -a 256 "$f"
done > weights_manifest.txt
shasum -a 256 weights_manifest.txt
```

最终的哈希就是写入事实 03 的内容。请同时包含每个分片的哈希和最终的
聚合哈希,以便用户在下载时即可对单独的分片进行验证。

## 接收方(您模型的使用者)应了解什么

一次附带 MYRIAM 密封发布的模型发布让用户可以:

1. **验证所下载的权重**:重新计算哈希并与事实 03 进行比对。
2. **验证模型卡未被事后追改**:对照已密封的事实 04 与实验室当前公开
   发布的版本。
3. **验证评测分数即为发布时的分数**:将实验室当前的评测声明与已密封
   的事实 05 比对。
4. **就部署前承诺向实验室追究问责**:将实际行为与已密封的事实 06 进行
   比对。

若用户发现存在不一致,即获得了关于该不一致的密码学证据。他们可以将
该证据公开发布;原始的密封发布即为锚点。

## 本套件**不是**什么

- 不是真正进行安全审查的替代品。密封是安全审查的 *记录*,而非审查
  本身。
- 不是真正进行评测的替代品。密封锚定您所产出的分数;它并不验证方法论。
- 不是模型安全的保证。密封记录的是实验室在发布时对模型作出的声明;
  它并不为这些声明背书。
- 不能防御下游用户对模型的滥用。

## 关于对抗性微调的说明

AI 安全领域的一项常见担忧:有人对您的模型进行微调以去除安全行为,
并将其再分发。密封无法阻止这件事。密封能做的是:

- 锚定 *原始* 模型本应做什么(事实 04)。
- 锚定原始评测分数(事实 05)。
- 提供一个参考点,可用来度量对抗性微调相对于原版的偏离。

遇到第三方再分发版本的用户,可以验证其权重是否与原始的已密封哈希
相匹配。若不匹配,用户即可得知自己拿到的是非原版模型,且原本归属
于该实验室的任何安全声明对其不适用。

## 结语

密封一次模型发布对实验室而言只是少量额外工作:约 10 分钟用于计算哈希
和运行密封脚本,再加上一个 30 秒的决定 —— 决定要纳入哪些部署前承诺。
它所增加的问责是非对称的:对诚实的实验室代价微小,对原本可能悄然漂移
其声明的实验室则构成强约束。

请派生它,修改它。承重的是数学本身,而非任何具体实现。
