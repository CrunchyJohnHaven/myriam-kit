# کیت تکرارپذیری علمی

*ترجمه فارسی — برای نسخه اصلی انگلیسی، [README.md](README.md) را ببینید*

نمونه‌ای آماده از myriam-kit برای گروه‌های پژوهشی، مجلات و نهادهای متمرکز بر تکرارپذیری که می‌خواهند یک مانیفست تکرارپذیری انکرشده در Bitcoin و دارای شواهد دستکاری را در کنار یک مقاله یا preprint منتشر کنند.

## آنچه این کیت تولید می‌کند

یک محصول منتشرشدهٔ مهر و موم‌شده که برای یک مطالعه شامل موارد زیر است:

1. شناسایی مطالعه (عنوان مقاله، نویسندگان، نهاد، تأمین‌کنندگان مالی، تعارض منافع)
2. **هش SHA-256 آرشیو دادهٔ خام** — مجموعه‌دادهٔ متعارف
3. **هش خط لولهٔ تحلیل** — اسکریپت + دایجست تصویر Docker + وابستگی‌های pin شده
4. **هش بستهٔ خروجی مورد انتظار** — آنچه یک تکرارکنندهٔ وفادار باید تولید کند
5. پروتکل تکرار + تعهدات انتشار پس از نتیجه
6. اصول‌گر(ای) امضاکننده

این مهر و موم برای مرحلهٔ **پیش از تحلیل** در نظر گرفته شده است: داده‌ها فریز شده، خط لولهٔ پردازش نهایی شده، خروجی مورد انتظار محاسبه شده، همهٔ این‌ها پیش از مشاهدهٔ هرگونه نتیجه به‌منظور نوشتن مقاله.

## آنچه این کیت در برابرش دفاع می‌کند

**حالت‌های شکستی که مهر و موم آن‌ها را دستگیر می‌کند:**

- **p-hacking**: تغییرات پس از تحلیل پس از دیدن نتایج از طریق diff بین FACT 04 مهر و موم‌شده و خط لولهٔ منتشرشده قابل تشخیص است.
- **HARKing** (فرضیه‌پردازی پس از شناخت نتایج): فرضیهٔ پیش‌ثبت‌شده در FACT 02 از نظر بایت پایدار است.
- **دستکاری داده پس از مقاله**: هش‌گیری مجدد آرشیو داده در برابر FACT 03 هرگونه تغییر را آشکار می‌کند.
- **رانش خط لوله**: هش‌گیری مجدد خط لولهٔ تحلیل در برابر FACT 04 تغییرات خاموشانه را دستگیر می‌کند.
- **بازی با نویسندگی**: افزودن یا حذف خاموشانهٔ نویسندگان پس از انتشار در برابر FACT 02 مهر و موم‌شده قابل تشخیص می‌شود.

**حالت‌های شکستی که پوشش داده نمی‌شوند:**

- تقلب در جمع‌آوری دادهٔ خام (مهر و موم، داده‌ها را در نقطهٔ مهر و موم انکر می‌کند، نه در نقطهٔ جمع‌آوری)
- عدم تکرارپذیری زیستی / تجربی (مهر و موم فقط تکرارپذیری محاسباتی را انکر می‌کند)
- مسائل اندازهٔ نمونه یا اعتبار جمعیتی (این‌ها سؤالات طراحی مطالعه هستند)
- درستی ادعای زیربنایی (مهر و موم آنچه را ادعا شده انکر می‌کند، نه اینکه آیا جهان با ادعا تطابق دارد)

## چه زمانی از این کیت استفاده کنیم

اگر در یکی از موقعیت‌های زیر هستید، مهر و موم را در نظر بگیرید:

- شما یک گروه پژوهشی هستید که تحلیلی پر-اهمیت (کارآزمایی بالینی، ارزیابی سیستم هوش مصنوعی، مطالعهٔ همانندسازی) اجرا می‌کنید و یکپارچگی فارنزیک فراتر از مهر زمان OSF / GitHub می‌خواهید
- شما یک پژوهشگر تازه‌کار هستید که حفاظت اعتباری نامتقارن می‌خواهد: پیش-تحلیل مهر و موم‌شده به این معناست که نقد «آن‌ها p-hacking کردند» در آینده یک ابطال‌گر عمومی خواهد داشت
- شما یک سردبیر مجله یا سرپرست پروژهٔ همانندسازی هستید و می‌خواهید پیش-ثبت در سطح رمزنگاری را برای زیرمجموعه‌ای از مقالات تشویق کنید
- شما روی تحلیلی کار می‌کنید که در آن دستکاری‌های پس از تحلیل عواقب میدانی معنادار دارند (روان‌شناسی، اقتصاد رفتاری، پژوهش پزشکی)

## سابقهٔ مرگ‌ومیر / آسیب تاریخی

ارتباط این کیت با مرگ‌ومیر غیرمستقیم است:

- **عدم تکرارپذیری پژوهش‌های سلول‌های بنیادی / سرطان:** چندین مقاله از طریق کارآزمایی‌های بالینی نادرست‌جهت‌گیری‌شده به آسیب بیمار منجر شده‌اند. پیش-ثبت مهر و موم‌شدهٔ خط لولهٔ اصلی، مبنای مقایسه را انکر می‌کرد.
- **مناقشهٔ اثربخشی Tamiflu:** بررسی Cochrane تا حدی به این دلیل سال‌ها به طول انجامید که دسترسی به دادهٔ خام مورد مناقشه بود. هش‌های دادهٔ مهر و موم‌شده در زمان انتشار، اختلافات دسترسی به داده را بلافاصله قابل حل می‌کرد.
- **بحران عمومی عدم تکرارپذیری:** هزینهٔ مرگ‌ومیر در سطح حوزه از پژوهش زیست‌پزشکی غیرقابل تکرار قابل توجه است (Begley & Ellis در سال ۲۰۱۲ برآورد کردند که بیش از ۵۰٪ یافته‌های پیش‌بالینی غیرقابل تکرارند). انکر کردن رمزنگاری یک اهرم ساختاری است.

## چگونه این کیت را fork کنیم

```bash
cp -r ~/Genesis/myriam-kit/examples/scientific-reproducibility ~/my-paper-seal
cd ~/my-paper-seal

# CRITICAL: do this BEFORE seeing any results
shasum -a 256 raw_data_v0.tar.gz       # compute the canonical data hash
shasum -a 256 analysis.py              # hash your pipeline
docker buildx imagetools inspect <image> --format '{{.Manifest.Digest}}'

$EDITOR facts/*.txt                    # populate with real values
bash build/seal.sh                     # seal BEFORE running the analysis
python3 build/verify.py                # confirm

# Now run the analysis. Hash the output and confirm matches FACT 05.
# If it doesn't match, the pipeline is non-deterministic; fix and re-seal.
```

## یکپارچگی با زیرساخت‌های پژوهشی موجود

- **arXiv / bioRxiv / medRxiv**: URL مهر MYRIAM یا CID IPFS را در چکیده یا مواد تکمیلی قرار دهید
- **OSF**: محصول مهر و موم را در کنار فرم پیش-ثبت معمولی به پروژهٔ OSF پیوست کنید
- **AsPredicted**: URL مهر MYRIAM را در سند پیش-ثبت قرار دهید
- **GitHub**: هش commit خط لولهٔ تحلیل چیزی است که FACT 04 انکر می‌کند؛ GitHub منبع را فراهم می‌کند، MYRIAM بایت‌ها را انکر می‌کند
- **Zenodo / Figshare / OpenAIRE**: دادهٔ خام را دپوزیت کنید، سپس هش دپوزیت را مهر و موم کنید؛ مهر و موم از نظر فارنزیک نسبت به فراداده‌های دپوزیت دشوارتر است که خاموشانه جایگزین شود
- **مواد تکمیلی مجله**: محصول مهر MYRIAM را به‌عنوان فایل تکمیلی در زمان ارسال قرار دهید

## آنچه این کیت جایگزین آن نمی‌شود

- تأییدیهٔ IRB / کمیتهٔ اخلاق
- داوری همتایان
- مطالعات همانندسازی (تکرارپذیری محاسباتی ≠ تکرارپذیری تجربی)
- توافق‌نامه‌های دسترسی به دادهٔ حساس

## یادداشتی دربارهٔ ترتیب نویسندگان و اعتبار

مهر و موم، فهرست نویسندگان را در زمان مهر و موم انکر می‌کند. اختلافات دربارهٔ نویسندگی که پس از انتشار بروز می‌کنند می‌توانند در برابر فهرست مهر و موم‌شده ارزیابی شوند. این یک اهرم پاسخگویی کوچک است — برای پژوهشگران تازه‌کاری که خاموشانه افزوده یا حذف می‌شوند مهم است.
