# 科学可复现性套件

一个开箱即用的 myriam-kit 范例,面向希望随论文或预印本一同发布**经
Bitcoin 锚定、可察觉篡改的可复现性清单**的研究团队、期刊与可复现性
导向的机构。

## 本套件产出什么

针对一项研究的一份密封发布,内容包含:

1. 研究标识(论文题目、作者、所属机构、资助方、利益冲突声明)
2. **原始数据归档的 SHA-256** —— 规范数据集
3. **分析流水线哈希** —— 脚本 + Docker 镜像摘要 + 锁定的依赖
4. **预期输出包哈希** —— 一个忠实的复现者应当得到的结果
5. 复现协议 + 结果出炉后的发布承诺
6. 签名负责人

该密封面向**分析前**阶段:数据冻结、流水线定稿、预期输出已计算完成
—— 全部都在为撰写论文而查看任何结果之前完成。

## 本套件防御什么

**该密封可捕获的失败模式:**

- **p-hacking(p 值操纵)**:看到结果后对分析的事后改动,可通过对
  比已密封的 FACT 04 与已发表的流水线进行检测。
- **HARKing**(Hypothesizing After Results are Known,在知道结果之
  后形成假设):FACT 02 中预注册的假设具有字节级稳定性。
- **论文发表后的数据操纵**:针对 FACT 03 对数据归档重新做哈希,可
  揭示任何修改。
- **流水线漂移**:针对 FACT 04 对分析流水线重新做哈希,可捕获悄然
  改动。
- **作者署名操弄**:论文发表之后悄然添加或移除作者的行为,可对照
  已密封的 FACT 02 被检测出来。

**未覆盖的失败模式:**

- 原始数据采集阶段的造假(密封锚定的是密封时点的数据,而非采集
  时点)
- 生物学 / 经验层面的不可复现性(密封仅锚定计算层面的可复现性)
- 样本量或总体效度问题(这些属于研究设计问题)
- 底层主张本身的真伪(密封锚定的是被宣称之事,而非世界是否与该
  宣称相符)

## 何时使用本套件

如果出现以下情形,可以考虑使用密封:

- 您是研究团队,正在执行高风险分析(临床试验、AI 系统评测、复现
  研究),希望获得超越 OSF / GitHub 时间戳之上的取证级完整性
- 您是青年研究者,希望获得不对称的声誉保护:已密封的分析前协议
  意味着未来任何"他们做了 p-hacking"的指控都会有一个公开的可证
  伪物
- 您是期刊编辑或复现项目负责人,希望对一部分论文鼓励采用密码学
  级别的预注册
- 您所做的分析属于事后微调会带来重大领域后果的领域(心理学、行
  为经济学、医学研究)

## 致死性 / 危害的历史先例

本套件与致死性的关联是间接的:

- **干细胞 / 癌症研究的不可复现性:**多篇论文曾通过误导性的临床
  试验导致患者受害。原始流水线的密封预注册本可锚定相应对照。
- **达菲(Tamiflu)疗效争议:**Cochrane 综述拖延多年,部分原因
  在于对原始数据的访问存在争议。发表时点的密封数据哈希,本可使
  数据访问争议立即可裁。
- **整体性的不可复现性危机:**不可复现的生物医学研究在领域层面
  造成的致死代价相当显著(Begley & Ellis 2012 估计 >50% 的临
  床前发现不可复现)。密码学锚定是其中一项结构性杠杆。

## 如何派生本套件

```bash
cp -r ~/Genesis/myriam-kit/examples/scientific-reproducibility ~/my-paper-seal
cd ~/my-paper-seal

# 关键:这一切必须在看到任何结果之前完成
shasum -a 256 raw_data_v0.tar.gz       # 计算规范数据哈希
shasum -a 256 analysis.py              # 对流水线做哈希
docker buildx imagetools inspect <image> --format '{{.Manifest.Digest}}'

$EDITOR facts/*.txt                    # 填入真实数值
bash build/seal.sh                     # 在执行分析之前完成密封
python3 build/verify.py                # 确认

# 现在执行分析。对输出做哈希并确认与 FACT 05 一致。
# 若不一致,说明流水线不具确定性;修复后重新密封。
```

## 与现有学术基础设施的对接

- **arXiv / bioRxiv / medRxiv**:在摘要或补充材料中包含 MYRIAM 密
  封 URL 或 IPFS CID
- **OSF**:将密封工件作为附件挂载到 OSF 项目,与常规预注册表单并列
- **AsPredicted**:在预注册文档中包含 MYRIAM 密封 URL
- **GitHub**:FACT 04 所锚定的是分析流水线的 commit 哈希;GitHub
  提供源代码,MYRIAM 锚定字节
- **Zenodo / Figshare / OpenAIRE**:存放原始数据后,对存放件的哈希
  进行密封;在取证层面,该密封比存放件的元数据更难被悄然替换
- **期刊补充材料**:在投稿时将 MYRIAM 密封工件作为补充文件一并提交

## 本套件**不**替代什么

- IRB / 伦理委员会审批
- 同行评议
- 复现研究(计算可复现性 ≠ 经验可复现性)
- 涉敏感数据的数据访问协议

## 关于作者顺序与署名的说明

密封锚定的是密封时点的作者列表。论文发表之后出现的署名争议,可
对照已密封的列表进行评估。这是一个小的问责杠杆 —— 对于那些会被
悄然加入或移除的青年研究者尤为重要。

## 关于密封边界的说明

请向自己以及您的读者明确:密封分析前协议会使 p-hacking **公开可
检测**,但密封**无法阻止不当分析,也无法修复人类动机问题**。一
位决心违反其已密封协议的研究者依然可以这么做;密封只能保证该违
反行为被留入记录。科学社会学层面 —— 职业激励、发表偏倚、经费压
力 —— 凌驾于任何密码学层之上。本套件提供的是一个可证伪物,而
非美德。
