# Швидкий старт

П'ять хвилин від клонування до опублікованої запечатаної публікації. Перевірено на macOS;
Linux має працювати; користувачам Windows знадобиться WSL.

## 0. Передумови

```bash
brew install python3 opentimestamps-client ipfs
python3 -m pip install --user certifi  # для SSL у macOS
```

## 1. Створення каркаса

```bash
git clone <wherever-this-lives> myriam-kit
cd myriam-kit
bash lib/myriam-init.sh my-first-publication
cd my-first-publication
```

## 2. Редагування фактів

```
facts/
├── 01_what_this_is.txt
├── 02_what_im_claiming.txt
└── 03_who_signs.txt
```

Кожен файл — це один факт. UTF-8. Закінчення рядків LF. Шаблон імені файлу —
`NN_short_name.txt`, де NN — число з провідним нулем. Вони сортуються
лексикографічно в дереві Меркла, тож 02 завжди йде після 01.

Підказка: формулюйте кожен факт як *подію іменування або декларації*, а не як
онтологічне твердження про світ. «Acme Corp заявила, що партія X
пройшла тест Y» — перевірне. «Партія X безпечна» — ні.

## 3. Запуск демона IPFS (одноразово на машину)

```bash
ipfs init                 # одноразово для користувача
ipfs daemon &             # у фоні, залиште запущеним
```

## 4. Запечатування

```bash
bash build/seal.sh
```

Це виконає такі дії:
1. Обчислить хеш кожного факту (SHA-256), обчислить корінь Меркла у стилі Bitcoin.
2. Поставить штамп на корені через OpenTimestamps з двома незалежними календарями
   (alice + finney за замовчуванням — перевизначається через `MYRIAM_CAL_A`/`MYRIAM_CAL_B`).
3. Закріпить теку `site/` в IPFS та виведе CID.

Усього: ~10 секунд із прогрітим демоном IPFS.

## 5. Редагування `site/index.html`

Замініть заповнювачі шаблону вашими реальними фактами, хешами та
коренем Меркла. Відображені хеші кожного факту повинні збігатися з файлами фактів,
інакше сторінка не пройде власну самоперевірку.

Для кожного факту скопіюйте тіло файлу .txt у блок `<div class="fact">`
і помістіть SHA-256 у `<div class="hash">` безпосередньо під ним.
Корінь Меркла йде до секції криптографічної печатки.

(Так, це можна було б генерувати автоматично. Це навмисно не зроблено: ручний
крок — це етап перегляду людиною. Протокол не захищає
від публікації того, чого ви не прочитали.)

## 6. Локальна перевірка

```bash
python3 build/verify.py
```

Очікуваний вивід:
```
OK  merkle root matches: <hex>
OK  N leaf hashes match
--- ots verify merkle_root.txt.ots ---
Pending confirmation in Bitcoin blockchain
```

«Pending» — це нормально — календар OTS оновить атестацію
до повної атестації Bitcoin протягом кількох годин.

## 7. Розгортання

Оберіть будь-що з:

- **Cloudflare Pages**: панель → Pages → Direct upload → перетягніть `site/`
- **Netlify**: `netlify deploy --dir=site --prod`
- **GitHub Pages**: `git init` у `site/`, push у репозиторій, увімкніть Pages
- **S3**: `aws s3 sync site/ s3://your-bucket/ --delete` (з увімкненим статичним хостингом)
- **Будь-який простий VPS**: `scp -r site/* user@host:/var/www/html/`

CID з IPFS, отриманий на кроці 4, дає вам додаткову URL-адресу за змістом:
`https://<CID>.ipfs.dweb.link/`. Використовуйте її.

## 8. Оновлення квитанції OTS (пізніше)

Протягом ~кількох годин після запечатування виконайте:

```bash
ots upgrade build/merkle_root.txt.ots
```

Це перетворює відкладену атестацію кожного календаря на повну атестацію Bitcoin,
яка посилається на конкретний блок. Повторно розгорніть файл `.ots` на ваших
хостах після оновлення.

## 9. Ось і все

Ви опублікували невеликий набір фактів так, щоб будь-яка третя сторона могла:

- Отримати всі факти через звичайний HTTP з вашого хоста
- Перерахувати корінь Меркла з байтів фактів
- Порівняти з вашим опублікованим коренем
- Переконатися, що корінь був закомічений до Bitcoin у час штампу OTS
- Незалежно отримати ті ж факти через IPFS за опублікованим CID
- Підтвердити, що всі хеші збігаються

Якщо будь-який із цих кроків не вдається, відбулося втручання. Читачеві не потрібно
довіряти вам. Йому достатньо просто перевірити математику.
