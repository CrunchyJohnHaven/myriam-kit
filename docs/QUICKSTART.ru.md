# Быстрый старт

Пять минут от клонирования до запечатанной публикации. Протестировано на macOS;
на Linux должно работать; пользователям Windows понадобится WSL.

## 0. Предварительные требования

```bash
brew install python3 opentimestamps-client ipfs
python3 -m pip install --user certifi  # for macOS SSL
```

## 1. Каркас

```bash
git clone <wherever-this-lives> myriam-kit
cd myriam-kit
bash lib/myriam-init.sh my-first-publication
cd my-first-publication
```

## 2. Отредактируйте факты

```
facts/
├── 01_what_this_is.txt
├── 02_what_im_claiming.txt
└── 03_who_signs.txt
```

Каждый файл — один факт. UTF-8. Окончания строк LF. Шаблон имени файла —
`NN_short_name.txt`, где NN — двузначный индекс с ведущим нулём. Файлы
сортируются лексикографически в дереве Меркла, поэтому 02 всегда идёт
после 01.

Совет: формулируйте каждый факт как *событие именования или декларации*,
а не как онтологическое утверждение о мире. «Acme Corp заявила, что
партия X прошла испытание Y» — проверяемо. «Партия X безопасна» —
непроверяемо.

## 3. Запустите демон IPFS (один раз на машину)

```bash
ipfs init                 # one-time per user
ipfs daemon &             # background, leave running
```

## 4. Запечатайте

```bash
bash build/seal.sh
```

Это сделает следующее:
1. Захэширует каждый факт (SHA-256), вычислит корень Меркла в стиле Bitcoin.
2. Проставит временную метку на корень через OpenTimestamps с использованием
   двух независимых календарей (alice + finney по умолчанию — переопределяется
   через `MYRIAM_CAL_A`/`MYRIAM_CAL_B`).
3. Закрепит каталог `site/` в IPFS и выведет CID.

Итого: ~10 секунд при разогретом демоне IPFS.

## 5. Отредактируйте `site/index.html`

Замените заполнители шаблона на ваши настоящие факты, хэши и корень Меркла.
Отображаемые хэши по фактам должны соответствовать файлам каждого факта,
иначе страница не пройдёт собственную самопроверку.

Для каждого факта скопируйте тело файла .txt в блок
`<div class="fact">` и поместите SHA-256 в `<div class="hash">`
непосредственно под ним. Корень Меркла помещается в секцию
криптографической печати.

(Да, это можно было бы автоматически сгенерировать. Это сделано не так,
намеренно: ручной шаг — это шаг человеческой проверки. Протокол не
защищает от публикации того, что вы не прочитали.)

## 6. Проверьте локально

```bash
python3 build/verify.py
```

Ожидаемый вывод:
```
OK  merkle root matches: <hex>
OK  N leaf hashes match
--- ots verify merkle_root.txt.ots ---
Pending confirmation in Bitcoin blockchain
```

«Pending» — это нормально: календарь OTS обновит аттестацию до полной
аттестации Bitcoin в течение нескольких часов.

## 7. Разверните

Выберите любой из вариантов:

- **Cloudflare Pages**: панель → Pages → Direct upload → перетащите `site/`
- **Netlify**: `netlify deploy --dir=site --prod`
- **GitHub Pages**: `git init` в `site/`, push в репозиторий, включите Pages
- **S3**: `aws s3 sync site/ s3://your-bucket/ --delete` (со включённым статическим хостингом)
- **Любой простой VPS**: `scp -r site/* user@host:/var/www/html/`

CID IPFS из шага 4 даёт вам дополнительный URL с адресацией по содержимому:
`https://<CID>.ipfs.dweb.link/`. Используйте его.

## 8. Обновите квитанцию OTS (позже)

В течение ~нескольких часов после запечатывания выполните:

```bash
ots upgrade build/merkle_root.txt.ots
```

Это преобразует ожидающую аттестацию каждого календаря в полную аттестацию
Bitcoin, ссылающуюся на конкретный блок. После обновления повторно разверните
файл `.ots` на ваших хостах.

## 9. Вот и всё

Вы опубликовали небольшое множество фактов так, что любая третья сторона
может:

- Получить все факты по обычному HTTP с вашего хоста
- Перевычислить корень Меркла из байтов фактов
- Сравнить с вашим опубликованным корнем
- Убедиться, что корень был закреплён в Bitcoin в момент, указанный
  меткой OTS
- Независимо получить те же факты через IPFS по опубликованному CID
- Подтвердить совпадение всех хэшей

Если какой-либо из этих шагов не пройдёт, имело место вмешательство.
Читателю не нужно доверять вам. Ему нужно лишь произвести вычисления.
