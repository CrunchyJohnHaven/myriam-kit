# Спецификация протокола MYRIAM-kit

## 0. Назначение

Этот документ задаёт минимальный протокол для публикации небольшого
множества текстовых фактов в форме с обнаружимыми изменениями и
криптографической проверяемостью, которая переживает удаление с любого
отдельного хостинг-провайдера.

Это обобщение протокола, впервые развёрнутого на
[disclosure.host](https://disclosure.host) под названием «MYRIAM v1».

## 1. Цели, которые не ставятся

- Хранение секретов. Всё публично по построению.
- Большие наборы данных. Свыше ~20 фактов протокол имеет неверную форму.
- Публикация в реальном времени. Каждая публикация — это дискретное
  событие запечатывания.
- Отзыв. Протокол работает только на добавление на уровне версии.

## 2. Модель факта

ФАКТ — это UTF-8-текстовый файл со следующими свойствами:

- Шаблон имени файла: `NN_short_name.txt`, где NN — двузначный индекс
  с ведущим нулём, начинающийся с 01.
- Каноническая форма: байты как есть, без нормализации.
- Хэш факта: шестнадцатеричное представление в нижнем регистре
  `SHA-256(file_bytes)`.

МНОЖЕСТВО фактов — это упорядоченный список всех файлов, соответствующих
шаблону, в каталоге `facts/`, отсортированных лексикографически.

Ограничение: тело каждого факта СЛЕДУЕТ формулировать как *событие
именования* или *событие декларации*, относящееся к конкретной стороне.
Онтологические утверждения («луна — это камень») подрывают порог
проверяемости; проверяемо только событие публикации, а не само
утверждение. «X заявил Y» — правильная форма. «Y истинно» — нет.

## 3. Корень Меркла

В стиле Bitcoin, как в v0 развёрнутого протокола MYRIAM:

1. Листья: хэши SHA-256 по каждому факту в порядке множества фактов.
2. Если уровень имеет нечётное число узлов, продублируйте последний узел.
3. Попарно хэшируйте соседние узлы:
   `parent = SHA-256(left_bytes || right_bytes)`, где `left_bytes` и
   `right_bytes` — сырые 32-байтные значения хэшей (а не их
   шестнадцатеричные представления).
4. Повторяйте, пока не останется один узел. Это и есть корень Меркла.

Корень публикуется в шестнадцатеричном виде в нижнем регистре без
завершающего перевода строки.

### 3.1 Тестовые векторы

| Вектор | Листья (хэшируются UTF-8 байты) | Ожидаемый корень (hex) |
|---|---|---|
| A | `"a"` | `ca978112ca1bbdcafac231b39a23dc4da786eff8147c4e72b9807785afee48bb` |
| B | `"a"`, `"b"` | `e5a01fee14e0ed5c48714f22180f25ad8365b53f9779f79dc4a3d7e93963f94a` |
| C | `"a"`, `"b"`, `"c"` | `d31a37ef6ac14a2db1470c4316beb5592e6afd4465022339adafda76a18ffabe` |

Любая совместимая реализация ДОЛЖНА воспроизводить эти векторы. Они
выведены из реального `lib/build.py` и могут быть повторно сгенерированы
этим скриптом.

## 4. Закрепление в Bitcoin

Корень Меркла закрепляется в блокчейне Bitcoin через
[OpenTimestamps](https://opentimestamps.org). Политика по умолчанию:

> Проставлять временную метку как минимум через двух операторов
> календарей с независимым административным контролем.

Квитанция, удовлетворяющая этой политике, проверяема любой стороной,
способной выполнить `ots verify <receipt>.ots` и достичь пути обновления
любого из календарей.

Пара операторов по умолчанию, поставляемая в `lib/seal.sh`:

- `alice.btc.calendar.opentimestamps.org` — Peter Todd
- `finney.calendar.eternitywall.com` — Eternity Wall

Реализации МОГУТ использовать других операторов. Политика разнообразия
свидетелей (≥2 независимых операторов) является нормативной;
конкретные операторы — нет.

## 5. Зеркалирование с адресацией по содержимому

Каталог `site/`, содержащий опубликованную страницу и файлы фактов,
СЛЕДУЕТ добавить в IPFS:

```
ipfs add -r --cid-version=1 -Q site/
```

Получаемый CIDv1 (кодек dag-pb) — это контентный адрес публикации. Он
детерминистичен от байтового содержимого `site/`. Любое изменение любого
файла в `site/` приводит к другому CID.

Закрепление в дополнительных сервисах помимо локального узла публикатора
настоятельно рекомендуется для долговечности; данная спецификация не
требует конкретных сервисов.

## 6. Версионирование

Протокол **работает только на добавление на уровне версии**. Любое
изменение требует новой запечатанной версии:

- Новая (или неизменная) редакция спецификации
- Новое множество фактов, которое МОЖЕТ наследовать неизменные факты из
  предыдущих версий
- Новый корень Меркла
- Новая квитанция OpenTimestamps
- Новый CID IPFS

Все предыдущие версии остаются опубликованными по версионированным URL.
Изменение любого факта v(N) требует v(N+1), причём обе версии остаются
видимыми и повторно проверяемыми.

## 7. Проверка

Любая третья сторона может проверить целостность публикации, выполнив
следующее:

1. Загрузить файлы фактов и пересчитать хэши SHA-256 по каждому факту.
2. Пересчитать корень Меркла согласно §3 и сравнить с опубликованным
   корнем.
3. Проверить квитанцию OpenTimestamps относительно блокчейна Bitcoin
   как минимум для полной аттестации одного календаря.
4. Независимо пересчитать CID IPFS через
   `ipfs add -r --cid-version=1` и сравнить с опубликованным CID.

`lib/verify.py` выполняет шаги 1–3 без внешних зависимостей, кроме
стандартной библиотеки Python и клиента `ots`. Шаг 4 требует клиент
`ipfs`.

## 8. Порог отказа

Публикация НЕ ДОЛЖНА включать факт, который:

- Идентифицирует частную третью сторону, не давшую согласия на
  упоминание её имени.
- Представляет гипотезу как подтверждение.
- Не может быть сведён к проверяемому событию именования/декларации
  или к криптографическому артефакту.
- Имеет разумно ожидаемое увеличение числа предотвратимых смертей в
  результате публикации.

Четвёртое правило — несущее. Оно применяется на шаге человеческой
проверки публикатором. На уровне протокола порога отказа не существует;
протокол публикует те байты, которые ему передали. Ответственность за
этот порог несёт публикатор.

## 9. Честное изложение ограничений

- **Обнаружимость, а не невозможность.** Протокол делает вмешательство
  обнаружимым, а не невозможным. Решительный атакующий по-прежнему
  может подменить публикацию другой и убедить часть читателей, что это
  оригинал. Ценность протокола в том, что подмена становится
  криптографически доказуемой для любого, кто проведёт проверку.
- **Нет защиты от публикации неправды.** Если публикатор запечатывает
  факт, гласящий «X произошло», а X не происходило, протокол
  запечатывает ложное утверждение с той же точностью, с какой
  запечатывает истинные. Протокол свидетельствует о том, что было
  *опубликовано*, а не о том, что *истинно*.
- **Подпись одной стороной.** v1 развёрнутого MYRIAM использует одного
  подписывающего принципала. Многосторонняя подпись — это работа на
  будущее.
- **Долговечность закрепления.** Закрепление в IPFS только на узле
  публикатора обеспечивает «доступность, пока мой ноутбук включён +
  оппортунистическое кэширование шлюзами». Долгосрочная долговечность
  требует избыточных закреплений в независимых сервисах.

Эти ограничения изложены здесь, чтобы ни один читатель не предполагал,
что протокол даёт больше, чем он на самом деле даёт.
