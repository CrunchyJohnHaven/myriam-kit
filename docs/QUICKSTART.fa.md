# شروع سریع

پنج دقیقه از clone تا انتشار مهر و موم‌شده. روی macOS آزمایش شده است؛ روی Linux باید کار کند؛ کاربران Windows به WSL نیاز خواهند داشت.

## ۰. پیش‌نیازها

```bash
brew install python3 opentimestamps-client ipfs
python3 -m pip install --user certifi  # for macOS SSL
```

## ۱. ساخت اسکلت (Scaffold)

```bash
git clone <wherever-this-lives> myriam-kit
cd myriam-kit
bash lib/myriam-init.sh my-first-publication
cd my-first-publication
```

## ۲. ویرایش واقعیت‌ها

```
facts/
├── 01_what_this_is.txt
├── 02_what_im_claiming.txt
└── 03_who_signs.txt
```

هر فایل یک واقعیت است. UTF-8. پایان خط LF. الگوی نام فایل
`NN_short_name.txt` است که در آن NN عدد دو رقمی صفر-پُرشده است. این فایل‌ها
به‌صورت لغت‌نامه‌ای در درخت Merkle مرتب می‌شوند، بنابراین ۰۲ همیشه بعد از ۰۱ می‌آید.

نکته: هر واقعیت را به‌صورت یک *رویداد نام‌گذاری یا اعلان* بیان کنید، نه به‌عنوان یک
ادعای وجودی دربارهٔ جهان. «Acme Corp اعلام کرده است که دستهٔ X
آزمون Y را گذرانده است» قابل راستی‌آزمایی است. «دستهٔ X ایمن است» نیست.

## ۳. راه‌اندازی دیمن IPFS (یک‌بار به‌ازای هر دستگاه)

```bash
ipfs init                 # one-time per user
ipfs daemon &             # background, leave running
```

## ۴. مهر و موم

```bash
bash build/seal.sh
```

این کار:
۱. هر واقعیت را هش می‌کند (SHA-256)، و ریشهٔ Merkle به سبک Bitcoin را محاسبه می‌کند.
۲. ریشه را با OpenTimestamps از طریق دو تقویم مستقل مهر می‌زند
   (به‌طور پیش‌فرض alice + finney — با `MYRIAM_CAL_A`/`MYRIAM_CAL_B` قابل override).
۳. دایرکتوری `site/` را در IPFS پین می‌کند و CID را چاپ می‌کند.

مجموع: ~۱۰ ثانیه با یک دیمن IPFS گرم.

## ۵. ویرایش `site/index.html`

placeholderهای الگو را با واقعیت‌ها، هش‌ها و ریشهٔ Merkle واقعی خود
جایگزین کنید. هش‌های نمایش‌داده‌شدهٔ هر واقعیت باید با فایل‌های هر واقعیت مطابقت داشته باشند،
وگرنه صفحه در راستی‌آزمایی خود شکست می‌خورد.

برای هر واقعیت، بدنهٔ فایل .txt را در یک بلاک `<div class="fact">`
کپی کنید و SHA-256 را در یک `<div class="hash">` درست زیر آن قرار دهید.
ریشهٔ Merkle در بخش مهر رمزنگاری قرار می‌گیرد.

(بله، این می‌تواند به‌صورت خودکار تولید شود. عمداً این کار انجام نشده است: مرحلهٔ
دستی همان مرحلهٔ بازبینی انسانی است. پروتکل از انتشار چیزی که
نخوانده‌اید محافظت نمی‌کند.)

## ۶. راستی‌آزمایی محلی

```bash
python3 build/verify.py
```

خروجی مورد انتظار:
```
OK  merkle root matches: <hex>
OK  N leaf hashes match
--- ots verify merkle_root.txt.ots ---
Pending confirmation in Bitcoin blockchain
```

«Pending» (در انتظار) عادی است — تقویم OTS ظرف چند ساعت گواهی را
به یک گواهی کامل Bitcoin ارتقا می‌دهد.

## ۷. استقرار

هرکدام را می‌توانید انتخاب کنید:

- **Cloudflare Pages**: داشبورد → Pages → آپلود مستقیم → کشیدن `site/`
- **Netlify**: `netlify deploy --dir=site --prod`
- **GitHub Pages**: `git init` در `site/`، push به مخزن، فعال‌سازی Pages
- **S3**: `aws s3 sync site/ s3://your-bucket/ --delete` (با میزبانی استاتیک روشن)
- **هر VPS ساده‌ای**: `scp -r site/* user@host:/var/www/html/`

CID مربوط به IPFS از مرحلهٔ ۴ یک URL محتوا-محور اضافی به شما می‌دهد:
`https://<CID>.ipfs.dweb.link/`. از آن استفاده کنید.

## ۸. ارتقای رسید OTS (بعداً)

ظرف ~چند ساعت پس از مهر و موم کردن، اجرا کنید:

```bash
ots upgrade build/merkle_root.txt.ots
```

این دستور گواهی در حال انتظار هر تقویم را به یک گواهی کامل Bitcoin
که به یک بلاک خاص ارجاع می‌دهد تبدیل می‌کند. پس از ارتقا، فایل `.ots` را دوباره
روی هاست‌های خود مستقر کنید.

## ۹. تمام شد

شما مجموعه‌ای کوچک از واقعیت‌ها را به گونه‌ای منتشر کرده‌اید که هر شخص ثالثی می‌تواند:

- همهٔ واقعیت‌ها را از طریق HTTP ساده از هاست شما fetch کند
- ریشهٔ Merkle را از بایت‌های واقعیت‌ها دوباره محاسبه کند
- آن را با ریشهٔ منتشرشدهٔ شما مقایسه کند
- راستی‌آزمایی کند که ریشه در زمان مهرخوردهٔ OTS به Bitcoin متعهد شده است
- همان واقعیت‌ها را به‌طور مستقل از طریق IPFS با CID منتشرشده fetch کند
- مطابقت همهٔ هش‌ها را تأیید کند

اگر هر یک از این مراحل شکست بخورد، دستکاری رخ داده است. خواننده
مجبور نیست به شما اعتماد کند. فقط باید ریاضیات را انجام دهد.
