# MYRIAM-kit પ્રોટોકોલ સ્પેસિફિકેશન

*ગુજરાતી અનુવાદ — મૂળ અંગ્રેજી સંસ્કરણ માટે, જુઓ [SPEC.md](SPEC.md)*

## 0. હેતુ

આ દસ્તાવેજ ટેક્સ્ટ તથ્યોના નાના સમૂહને ચેડાં-સ્પષ્ટ, ક્રિપ્ટોગ્રાફિક
રીતે ચકાસી શકાય તેવા સ્વરૂપમાં પ્રકાશિત કરવા માટેનો એક ન્યૂનતમ પ્રોટોકોલ
સ્પષ્ટ કરે છે જે કોઈપણ એક હોસ્ટિંગ પ્રદાતાની ટેકડાઉનથી ટકી શકે.

તે પ્રથમ વખત [disclosure.host](https://disclosure.host) પર "MYRIAM v1"
નામ હેઠળ જમાવવામાં આવેલ પ્રોટોકોલનું સામાન્યીકરણ છે.

## 1. બિન-લક્ષ્યો

- ગુપ્ત સંગ્રહ. બધું જ રચના દ્વારા જાહેર છે.
- મોટા ડેટાસેટ્સ. ~20 તથ્યો ઉપરાંત, પ્રોટોકોલ ખોટો આકાર છે.
- વાસ્તવિક-સમય પ્રકાશન. દરેક પ્રકાશન એક અલગ સીલિંગ ઘટના છે.
- પાછું ખેંચવું. પ્રોટોકોલ સંસ્કરણ સ્તરે ફક્ત-જોડાવા યોગ્ય છે.

## 2. તથ્ય મોડેલ

એક FACT એ UTF-8 ટેક્સ્ટ ફાઇલ છે જેમાં:

- ફાઇલનામ પેટર્ન: `NN_short_name.txt` જ્યાં NN એક શૂન્ય-પૅડેડ બે-અંકનો
  ઇન્ડેક્સ છે જે 01 થી શરૂ થાય છે.
- પ્રામાણિક સ્વરૂપ: બાઇટ્સ જેવા લખેલા છે તેવા, કોઈ સામાન્યીકરણ નહીં.
- પ્રતિ-તથ્ય હેશ: `SHA-256(file_bytes)` નો લોઅરકેસ hex.

તથ્ય SET એ `facts/` હેઠળ પેટર્ન સાથે મેળ ખાતી બધી ફાઇલોની ક્રમબદ્ધ
યાદી છે, જે શબ્દકોશીય ક્રમે સૉર્ટ કરેલ છે.

મર્યાદા: દરેક તથ્યનો મુખ્ય ભાગ SHOULD કોઈ ચોક્કસ પક્ષને જવાબદાર ઠેરવી
શકાય તેવી *નામકરણ ઘટના* અથવા *ઘોષણા ઘટના* તરીકે વ્યક્ત કરવો. ઑન્ટોલોજિકલ
દાવાઓ ("ચંદ્ર ખડક છે") ચકાસણીક્ષમતા ગેટને નબળો પાડે છે; ફક્ત પ્રકાશન
ઘટના ચકાસી શકાય તેવી છે, દાવો નહીં. "X એ Y જણાવ્યું છે" એ યોગ્ય સ્વરૂપ
છે. "Y સાચું છે" નથી.

## 3. Merkle મૂળ

Bitcoin-શૈલીનું, જમાવેલ MYRIAM પ્રોટોકોલના v0 માં છે તેમ:

1. પાંદડાઓ: તથ્ય-સમૂહ ક્રમમાં પ્રતિ-તથ્ય SHA-256 હેશ.
2. જો કોઈ સ્તર પર નોડ્સની વિષમ સંખ્યા હોય, તો છેલ્લા નોડની નકલ કરો.
3. અડોઅડ નોડ્સને જોડીમાં હેશ કરો:
   `parent = SHA-256(left_bytes || right_bytes)`,
   જ્યાં `left_bytes` અને `right_bytes` કાચા 32-બાઇટ હેશ મૂલ્યો છે
   (તેમના hex રજૂઆતો નહીં).
4. એક નોડ બાકી રહે ત્યાં સુધી પુનરાવર્તન કરો. તે Merkle મૂળ છે.

મૂળ ટ્રેઇલિંગ ન્યૂલાઇન વગર લોઅરકેસ hex તરીકે પ્રકાશિત થાય છે.

### 3.1 પરીક્ષણ વેક્ટર્સ

| Vector | Leaves (UTF-8 bytes hashed) | Expected root (hex) |
|---|---|---|
| A | `"a"` | `ca978112ca1bbdcafac231b39a23dc4da786eff8147c4e72b9807785afee48bb` |
| B | `"a"`, `"b"` | `e5a01fee14e0ed5c48714f22180f25ad8365b53f9779f79dc4a3d7e93963f94a` |
| C | `"a"`, `"b"`, `"c"` | `d31a37ef6ac14a2db1470c4316beb5592e6afd4465022339adafda76a18ffabe` |

કોઈપણ સુસંગત અમલીકરણ MUST આ વેક્ટર્સનું પુનઃઉત્પાદન કરે. તેઓ વાસ્તવિક
`lib/build.py` માંથી મેળવેલા છે અને તે સ્ક્રિપ્ટ સાથે ફરીથી જનરેટ કરી શકાય છે.

## 4. Bitcoin પ્રતિબદ્ધતા

Merkle મૂળ [OpenTimestamps](https://opentimestamps.org) દ્વારા Bitcoin
બ્લોકચેન માટે પ્રતિબદ્ધ થાય છે. મૂળભૂત નીતિ છે:

> સ્વતંત્ર વહીવટી નિયંત્રણ હેઠળના ઓછામાં ઓછા બે કૅલેન્ડર ઓપરેટરો સામે
> સ્ટેમ્પ કરો.

આ નીતિને સંતોષતી રસીદ કોઈપણ પક્ષ દ્વારા ચકાસી શકાય તેવી છે જે
`ots verify <receipt>.ots` ચલાવી શકે અને કોઈપણ એક કૅલેન્ડરના અપગ્રેડ
પાથ સુધી પહોંચી શકે.

`lib/seal.sh` માં શિપ થયેલ મૂળભૂત ઓપરેટર જોડી:

- `alice.btc.calendar.opentimestamps.org` — Peter Todd
- `finney.calendar.eternitywall.com` — Eternity Wall

અમલીકરણો MAY અલગ ઓપરેટરોનો ઉપયોગ કરે. સાક્ષી-વિવિધતા નીતિ (≥2 સ્વતંત્ર
ઓપરેટરો) આદર્શાત્મક છે; ચોક્કસ ઓપરેટરો નથી.

## 5. સામગ્રી-એડ્રેસ્ડ મિરરિંગ

પ્રકાશિત પૃષ્ઠ અને તથ્ય ફાઇલો ધરાવતી `site/` ડિરેક્ટરી SHOULD IPFS માં
ઉમેરવામાં આવે:

```
ipfs add -r --cid-version=1 -Q site/
```

પરિણામી CIDv1 (dag-pb codec) એ પ્રકાશનનું સામગ્રી સરનામું છે. તે `site/`
ની બાઇટ સામગ્રીમાંથી નિર્ધારક છે. `site/` હેઠળની કોઈપણ ફાઇલમાં કોઈપણ
ફેરફાર એક અલગ CID ઉત્પન્ન કરે છે.

પ્રકાશકના સ્થાનિક નોડ ઉપરાંતની વધારાની સેવાઓ પર પિન કરવાની ટકાઉપણા
માટે ભારપૂર્વક ભલામણ કરવામાં આવે છે; આ સ્પેસિફિકેશન ચોક્કસ સેવાઓ ફરજિયાત
કરતું નથી.

## 6. સંસ્કરણ

પ્રોટોકોલ **સંસ્કરણ સ્તરે ફક્ત-જોડાવા યોગ્ય** છે. દરેક ફેરફાર માટે એક
નવા સીલબંધ સંસ્કરણની જરૂર પડે છે:

- એક નવી (અથવા અપરિવર્તિત) સ્પેસિફિકેશન રિવિઝન
- એક નવો તથ્ય સમૂહ, જે MAY પૂર્વ સંસ્કરણોમાંથી તથ્યો અપરિવર્તિત વારસામાં લે
- એક નવું Merkle મૂળ
- એક નવી OpenTimestamps રસીદ
- એક નવો IPFS CID

બધા પૂર્વ સંસ્કરણો સંસ્કરણિત URLs પર પ્રકાશિત રહે છે. કોઈપણ v(N) તથ્યમાં
ફેરફાર માટે v(N+1) જરૂરી છે, જેમાં બંને સંસ્કરણો દૃશ્યમાન અને
ફરી-ચકાસી શકાય તેવા હોય.

## 7. ચકાસણી

કોઈપણ ત્રીજો પક્ષ પ્રકાશનની અખંડિતતા આ રીતે ચકાસી શકે:

1. તથ્ય ફાઇલો ડાઉનલોડ કરીને પ્રતિ-તથ્ય SHA-256 હેશની ફરીથી ગણતરી કરો.
2. §3 પ્રમાણે Merkle મૂળની ફરીથી ગણતરી કરો અને પ્રકાશિત મૂળ સાથે તુલના કરો.
3. ઓછામાં ઓછા એક કૅલેન્ડરના સંપૂર્ણ પ્રમાણીકરણ માટે Bitcoin બ્લોકચેન સામે
   OpenTimestamps રસીદ ચકાસો.
4. `ipfs add -r --cid-version=1` દ્વારા IPFS CID ની સ્વતંત્ર રીતે ફરીથી
   ગણતરી કરો અને પ્રકાશિત CID સાથે તુલના કરો.

`lib/verify.py` Python stdlib અને `ots` ક્લાયન્ટ ઉપરાંત કોઈ બાહ્ય
અવલંબન વગર પગલાં 1–3 કરે છે. પગલું 4 માટે `ipfs` ક્લાયન્ટ જરૂરી છે.

## 8. અસ્વીકાર ગેટ

એક પ્રકાશન MUST NOT એવા તથ્યનો સમાવેશ કરે જે:

- એક ખાનગી ત્રીજા પક્ષને ઓળખે જેણે નામ આપવા માટે સંમતિ આપી નથી.
- એક પૂર્વધારણાને પુષ્ટિ તરીકે રજૂ કરે.
- ચકાસી શકાય તેવી નામકરણ/ઘોષણા ઘટના અથવા ક્રિપ્ટોગ્રાફિક આર્ટિફેક્ટ સુધી
  ઘટાડી શકાય નહીં.
- પ્રકાશન પર નિવારી શકાય તેવા મૃત્યુમાં વાજબી અપેક્ષિત વધારો ધરાવે છે.

ચોથો નિયમ ભાર વહન કરનાર છે. તે પ્રકાશક દ્વારા માનવ સમીક્ષા પગલા પર
લાગુ કરાય છે. અસ્વીકાર ગેટનું કોઈ પ્રોટોકોલ-સ્તરનું પ્રવર્તન નથી;
પ્રોટોકોલ તેને જે પણ બાઇટ્સ આપવામાં આવે તે પ્રકાશિત કરે છે. પ્રકાશક
ગેટ માટે જવાબદાર છે.

## 9. મર્યાદાઓનું પ્રામાણિક નિવેદન

- **શોધક્ષમતા, અશક્યતા નહીં.** પ્રોટોકોલ ચેડાંને શોધી શકાય તેવા બનાવે
  છે, અશક્ય નહીં. એક નિર્ધારિત હુમલાખોર હજુ પણ એક અલગ પ્રકાશન બદલી
  શકે છે અને કેટલાક વાચકોને મનાવી શકે છે કે તે મૂળ છે. પ્રોટોકોલનું
  મૂલ્ય એ છે કે અવેજી ચકાસણી ચલાવનાર કોઈપણ માટે ક્રિપ્ટોગ્રાફિક રીતે
  દર્શાવી શકાય તેવી બને છે.
- **અસત્ય પ્રકાશિત કરવા સામે કોઈ સંરક્ષણ નહીં.** જો પ્રકાશક "X થયું"
  કહેતું તથ્ય સીલ કરે અને X ન બન્યું હોય, તો પ્રોટોકોલ ખોટા દાવાને
  એ જ વફાદારી સાથે સીલ કરે છે જે વફાદારી સાથે તે સાચાને સીલ કરે છે.
  પ્રોટોકોલ જે *પ્રકાશિત* થયું તેને પ્રમાણિત કરે છે, જે *સાચું* છે
  તેને નહીં.
- **એક-પક્ષીય સહી.** જમાવેલ MYRIAM ના v1 એક સહી કરનાર પ્રિન્સિપાલનો
  ઉપયોગ કરે છે. બહુ-પક્ષીય સહી ભવિષ્યનું કાર્ય છે.
- **પિનિંગ ટકાઉપણા.** ફક્ત પ્રકાશકના નોડ પર IPFS પિનિંગ "જ્યારે મારું
  લેપટોપ ચાલુ હોય ત્યારે ઉપલબ્ધ + તકવાદી ગેટવે કૅશિંગ" પ્રદાન કરે છે.
  લાંબા-ગાળાની ટકાઉપણા સ્વતંત્ર સેવાઓમાં વધારાની પિન જરૂરી છે.

આ મર્યાદાઓ અહીં જણાવેલી છે જેથી કોઈ વાચક માની ન લે કે પ્રોટોકોલ તે
આપે છે તેના કરતાં વધુ આપે છે.
