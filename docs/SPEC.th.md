# MYRIAM-kit Protocol Specification

*คำแปลภาษาไทย — สำหรับฉบับต้นฉบับภาษาอังกฤษ ดู [QUICKSTART.md](QUICKSTART.md) / [SPEC.md](SPEC.md)*

## 0. วัตถุประสงค์

เอกสารฉบับนี้กำหนดโพรโทคอลขั้นต่ำสำหรับการเผยแพร่ข้อเท็จจริงที่เป็น
ข้อความจำนวนน้อย ในรูปแบบที่ตรวจพบการแก้ไขได้ ตรวจสอบยืนยันได้ในเชิง
การเข้ารหัส และสามารถอยู่รอดได้แม้ผู้ให้บริการโฮสติ้งรายใดรายหนึ่งจะถูกระงับ

นี่คือการสรุปทั่วไป (generalization) ของโพรโทคอลที่ถูกนำมาใช้งานครั้งแรกที่
[disclosure.host](https://disclosure.host) ภายใต้ชื่อ "MYRIAM v1."

## 1. สิ่งที่ "ไม่" ใช่เป้าหมาย

- การเก็บความลับ ทุกสิ่งเป็นสาธารณะโดยโครงสร้าง
- ชุดข้อมูลขนาดใหญ่ เมื่อเกินประมาณ 20 ข้อเท็จจริง โพรโทคอลนี้คือรูปร่างที่ผิด
- การเผยแพร่แบบเรียลไทม์ การเผยแพร่แต่ละครั้งคือเหตุการณ์ผนึกที่เป็นเอกเทศ
- การถอนข้อความ โพรโทคอลนี้เป็น append-only ในระดับเวอร์ชัน

## 2. แบบจำลองข้อเท็จจริง (Fact model)

FACT คือไฟล์ข้อความ UTF-8 ที่มี:

- รูปแบบชื่อไฟล์: `NN_short_name.txt` โดย NN คือดัชนีสองหลักที่เติมศูนย์
  ด้านหน้า เริ่มต้นที่ 01
- รูปแบบ canonical: ไบต์ตามที่เขียน ไม่มีการ normalize
- แฮชต่อข้อเท็จจริง: เลขฐานสิบหกตัวพิมพ์เล็กของ `SHA-256(file_bytes)`

SET ของข้อเท็จจริงคือรายการตามลำดับของไฟล์ทั้งหมดที่ตรงกับรูปแบบ
ภายใต้ `facts/` เรียงแบบ lexicographic

ข้อจำกัด: เนื้อความของทุกข้อเท็จจริง SHOULD ถูกเรียบเรียงเป็น *เหตุการณ์การตั้งชื่อ*
หรือ *เหตุการณ์การประกาศ* ที่ระบุได้ว่ามาจากฝ่ายใดเฉพาะเจาะจง ข้อกล่าวอ้างเชิง
ontological ("ดวงจันทร์เป็นหิน") บั่นทอน gate ของการตรวจสอบยืนยัน เพราะ
มีแต่เหตุการณ์การเผยแพร่เท่านั้นที่ตรวจสอบยืนยันได้ ไม่ใช่ตัวข้อกล่าวอ้าง รูปแบบ
ที่ถูกต้องคือ "X ได้กล่าวว่า Y" ไม่ใช่ "Y เป็นความจริง"

## 3. ราก Merkle

รูปแบบ Bitcoin เช่นเดียวกับใน v0 ของโพรโทคอล MYRIAM ที่ deploy แล้ว:

1. ใบ (leaves): แฮช SHA-256 ต่อข้อเท็จจริง ตามลำดับของ fact set
2. ถ้าระดับใดมีจำนวน node เป็นเลขคี่ ให้ทำสำเนา node สุดท้ายซ้ำ
3. แฮชคู่ node ที่อยู่ติดกัน: `parent = SHA-256(left_bytes || right_bytes)`
   โดย `left_bytes` และ `right_bytes` เป็นค่าแฮชดิบขนาด 32 ไบต์
   (ไม่ใช่รูปแบบ hex ของพวกมัน)
4. ทำซ้ำจนเหลือ node เดียว นั่นคือราก Merkle

ราก Merkle เผยแพร่ในรูปแบบ hex ตัวพิมพ์เล็ก โดยไม่มี newline ปิดท้าย

### 3.1 Test vectors

| Vector | Leaves (UTF-8 bytes hashed) | Expected root (hex) |
|---|---|---|
| A | `"a"` | `ca978112ca1bbdcafac231b39a23dc4da786eff8147c4e72b9807785afee48bb` |
| B | `"a"`, `"b"` | `e5a01fee14e0ed5c48714f22180f25ad8365b53f9779f79dc4a3d7e93963f94a` |
| C | `"a"`, `"b"`, `"c"` | `d31a37ef6ac14a2db1470c4316beb5592e6afd4465022339adafda76a18ffabe` |

ทุกการ implement ที่สอดคล้อง MUST ผลิตเวกเตอร์เหล่านี้ซ้ำได้ พวกมัน
ได้มาจากไฟล์ `lib/build.py` จริง และสามารถ regenerate ได้ด้วยสคริปต์นั้น

## 4. การ commit สู่ Bitcoin

ราก Merkle ถูก commit เข้าสู่บล็อกเชน Bitcoin ผ่าน
[OpenTimestamps](https://opentimestamps.org) นโยบายเริ่มต้นคือ:

> ประทับเวลากับผู้ให้บริการ calendar อย่างน้อยสองรายภายใต้การควบคุม
> ดูแล (administrative control) ที่เป็นอิสระต่อกัน

ใบเสร็จที่ตอบสนองนโยบายนี้สามารถตรวจสอบยืนยันได้โดยฝ่ายใดก็ตามที่สามารถ
รัน `ots verify <receipt>.ots` และเข้าถึงเส้นทางการอัปเกรดของ calendar
อย่างน้อยหนึ่งราย

คู่ผู้ให้บริการเริ่มต้นที่แถมมาใน `lib/seal.sh`:

- `alice.btc.calendar.opentimestamps.org` — Peter Todd
- `finney.calendar.eternitywall.com` — Eternity Wall

การ implement MAY ใช้ผู้ให้บริการต่างออกไป นโยบายความหลากหลายของพยาน
(≥2 ผู้ให้บริการที่เป็นอิสระต่อกัน) มีผลเชิง normative ตัวผู้ให้บริการ
เฉพาะรายนั้นไม่ใช่

## 5. การมิเรอร์แบบอ้างอิงเนื้อหา (Content-addressed mirroring)

ไดเรกทอรี `site/` ที่บรรจุหน้าเว็บที่เผยแพร่และไฟล์ข้อเท็จจริง SHOULD ถูกเพิ่ม
เข้าสู่ IPFS:

```
ipfs add -r --cid-version=1 -Q site/
```

CIDv1 (โคเดก dag-pb) ที่ได้คือที่อยู่เชิงเนื้อหา (content address) ของสิ่งที่เผยแพร่
มันเป็น deterministic จากเนื้อหาไบต์ของ `site/` การแก้ไขใด ๆ ต่อไฟล์ใด ๆ
ภายใต้ `site/` จะผลิต CID ที่ต่างออกไป

แนะนำอย่างยิ่งให้ปักหมุดเพิ่มเติมไปยังบริการอื่นนอกเหนือจาก node ในเครื่อง
ของผู้เผยแพร่เพื่อความคงทน ข้อกำหนดฉบับนี้ไม่บังคับว่าต้องเป็นบริการใด

## 6. การทำเวอร์ชัน

โพรโทคอลนี้ **append-only ในระดับเวอร์ชัน** การเปลี่ยนแปลงทุกครั้ง
ต้องการเวอร์ชันที่ผนึกใหม่:

- การแก้ไขข้อกำหนด (specification revision) ใหม่หรือเดิม
- fact set ใหม่ ซึ่ง MAY สืบทอดข้อเท็จจริงโดยไม่เปลี่ยนแปลงจากเวอร์ชันก่อนหน้า
- ราก Merkle ใหม่
- ใบเสร็จ OpenTimestamps ใหม่
- CID ของ IPFS ใหม่

เวอร์ชันก่อนหน้าทุกเวอร์ชันยังคงเผยแพร่อยู่ที่ URL ที่ทำเวอร์ชันไว้ การเปลี่ยนแปลง
ใด ๆ ต่อข้อเท็จจริง v(N) ต้องการ v(N+1) โดยทั้งสองเวอร์ชันยังคงมองเห็นและ
ตรวจสอบยืนยันใหม่ได้

## 7. การตรวจสอบยืนยัน

บุคคลที่สามใด ๆ อาจตรวจสอบยืนยันความสมบูรณ์ของสิ่งที่เผยแพร่โดย:

1. ดาวน์โหลดไฟล์ข้อเท็จจริงและคำนวณแฮช SHA-256 ต่อข้อเท็จจริงใหม่
2. คำนวณราก Merkle ใหม่ตาม §3 และเทียบกับรากที่เผยแพร่
3. ตรวจสอบยืนยันใบเสร็จ OpenTimestamps กับบล็อกเชน Bitcoin
   สำหรับการยืนยันเต็มรูปแบบของ calendar อย่างน้อยหนึ่งราย
4. คำนวณ CID ของ IPFS ใหม่ด้วยตนเองผ่าน `ipfs add -r --cid-version=1`
   และเทียบกับ CID ที่เผยแพร่

`lib/verify.py` ทำขั้นตอนที่ 1–3 โดยไม่ต้องอาศัย dependency ภายนอก
นอกเหนือจาก stdlib ของ Python และ client `ots` ส่วนขั้นตอนที่ 4
ต้องใช้ client `ipfs`

## 8. Refusal gate

สิ่งที่เผยแพร่ MUST NOT มีข้อเท็จจริงที่:

- ระบุตัวบุคคลที่สามผู้เป็นเอกชนซึ่งไม่ได้ยินยอมให้มีการระบุชื่อ
- นำเสนอสมมติฐานว่าเป็นการยืนยัน (confirmation)
- ไม่สามารถถูกลดทอนให้กลายเป็นเหตุการณ์การตั้งชื่อ/การประกาศที่ตรวจสอบยืนยันได้
  หรือเป็นอาร์ติแฟกต์ทางการเข้ารหัส
- มีความคาดหวังตามสมเหตุสมผลว่าจะเพิ่มการเสียชีวิตที่ป้องกันได้เมื่อเผยแพร่

กฎข้อที่สี่คือกฎที่รับน้ำหนัก (load-bearing) มันถูกบังคับใช้ ณ ขั้นตอนการทบทวน
โดยมนุษย์ของผู้เผยแพร่ ไม่มีการบังคับใช้ refusal gate ในระดับโพรโทคอล
โพรโทคอลเผยแพร่ไบต์ใด ๆ ก็ตามที่ส่งให้มัน ผู้เผยแพร่เป็นผู้รับผิดชอบ gate นี้

## 9. คำแถลงข้อจำกัดอย่างซื่อตรง

- **ตรวจพบได้ ไม่ใช่ทำให้เป็นไปไม่ได้** โพรโทคอลทำให้การแก้ไข
  ตรวจพบได้ ไม่ใช่ทำให้เป็นไปไม่ได้ ผู้โจมตีที่มุ่งมั่นยังสามารถนำสิ่งที่
  เผยแพร่อันอื่นมาแทนที่และโน้มน้าวให้ผู้อ่านบางคนเชื่อว่ามันคือฉบับดั้งเดิม
  คุณค่าของโพรโทคอลคือการแทนที่นั้นจะกลายเป็นสิ่งที่พิสูจน์ได้ใน
  เชิงการเข้ารหัสต่อทุกคนที่รันการตรวจสอบ
- **ไม่มีการป้องกันการเผยแพร่ความเท็จ** ถ้าผู้เผยแพร่ผนึกข้อเท็จจริงที่ระบุว่า
  "X เกิดขึ้น" แต่ X ไม่ได้เกิดขึ้น โพรโทคอลก็ผนึกข้อกล่าวอ้างเท็จด้วย
  ความเที่ยงตรงเดียวกับที่ผนึกข้อกล่าวอ้างจริง โพรโทคอลรับรองสิ่งที่ถูก
  *เผยแพร่* ไม่ใช่สิ่งที่*เป็นความจริง*
- **การลงนามฝ่ายเดียว** v1 ของ MYRIAM ที่ deploy แล้วใช้ principal ที่ลงนาม
  เพียงรายเดียว การลงนามแบบหลายฝ่ายเป็นงานในอนาคต
- **ความคงทนของการปักหมุด** การปักหมุด IPFS ที่ node ของผู้เผยแพร่
  เพียงแห่งเดียวให้เพียง "พร้อมใช้งานในขณะที่ laptop ของฉันเปิดอยู่ +
  การ cache แบบฉวยโอกาสของ gateway" ความคงทนระยะยาวต้องการ
  การปักหมุดสำรองข้ามบริการที่เป็นอิสระต่อกัน

ข้อจำกัดเหล่านี้ระบุไว้ที่นี่เพื่อให้ผู้อ่านไม่สมมุติว่าโพรโทคอลส่งมอบสิ่งใด
มากไปกว่าที่มันส่งมอบจริง
