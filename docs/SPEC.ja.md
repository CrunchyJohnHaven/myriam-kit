# MYRIAM-kit プロトコル仕様

*日本語訳 — 英語の原文は [SPEC.md](SPEC.md) を参照*

## 0. 目的

本文書は、少数のテキスト事実(text facts)を、改ざんが検出可能で暗号的に検証可能、かついかなる単一ホスティング事業者によるテイクダウンも乗り越えて存続する形で公開するための最小限のプロトコルを規定します。

これは [disclosure.host](https://disclosure.host) において「MYRIAM v1」の名称で初めてデプロイされたプロトコルの一般化です。

## 1. 非目標

- 秘密の保管。すべては構造的に公開されます。
- 大規模データセット。事実が約 20 件を超える領域では、本プロトコルは適切な形状ではありません。
- リアルタイム公開。各公開は離散的な封緘イベントです。
- 撤回。本プロトコルはバージョン単位で追記専用(append-only)です。

## 2. 事実(fact)モデル

FACT(事実)は、以下を満たす UTF-8 テキストファイルです。

- ファイル名パターン: `NN_short_name.txt`。NN は 01 から始まるゼロ埋めの 2 桁インデックス。
- 正規形: 書かれたままのバイト列。正規化は行わない。
- 各事実のハッシュ: `SHA-256(file_bytes)` の小文字 16 進表記。

事実 SET(集合)は、`facts/` 配下のこのパターンに一致するすべてのファイルを辞書順にソートした順序付きリストです。

制約: すべての事実の本文は、特定の当事者に帰属可能な *命名イベント(naming event)* または *宣言イベント(declaration event)* として表現される **べきです(SHOULD)**。存在論的主張(「月は岩でできている」)は検証可能性ゲートを掘り崩します。検証可能なのは公開イベントだけであって、主張そのものではありません。「X は Y と述べた」が適切な形式です。「Y は真である」はそうではありません。

## 3. Merkle ルート

デプロイ済みの MYRIAM プロトコル v0 と同じく、Bitcoin 流に計算します。

1. 葉(leaves): 事実集合の順序での各事実の SHA-256 ハッシュ。
2. ある段(level)のノード数が奇数の場合、最後のノードを複製する。
3. 隣接ノードをペアにしてハッシュ化: `parent = SHA-256(left_bytes || right_bytes)`。ここで `left_bytes` と `right_bytes` は生の 32 バイトのハッシュ値であり、16 進表現ではありません。
4. ノードが 1 つになるまで繰り返す。それが Merkle ルートです。

ルートは小文字 16 進表記で、末尾の改行なしで公開されます。

### 3.1 テストベクトル

| ベクトル | 葉(ハッシュ化される UTF-8 バイト列) | 期待されるルート(hex) |
|---|---|---|
| A | `"a"` | `ca978112ca1bbdcafac231b39a23dc4da786eff8147c4e72b9807785afee48bb` |
| B | `"a"`, `"b"` | `e5a01fee14e0ed5c48714f22180f25ad8365b53f9779f79dc4a3d7e93963f94a` |
| C | `"a"`, `"b"`, `"c"` | `d31a37ef6ac14a2db1470c4316beb5592e6afd4465022339adafda76a18ffabe` |

準拠する実装は、これらのベクトルを再現 **しなければなりません(MUST)**。これらは実際の `lib/build.py` から導出されたものであり、同スクリプトで再生成できます。

## 4. Bitcoin コミットメント

Merkle ルートは [OpenTimestamps](https://opentimestamps.org) を介して Bitcoin ブロックチェーンにコミットされます。デフォルトのポリシーは以下のとおりです。

> 独立した管理下にあるカレンダーオペレーターを少なくとも 2 つ使ってスタンプすること。

このポリシーを満たすレシートは、`ots verify <receipt>.ots` を実行し、いずれかのカレンダーのアップグレードパスに到達できる任意の当事者によって検証可能です。

`lib/seal.sh` に同梱されるデフォルトのオペレーター対は以下のとおりです。

- `alice.btc.calendar.opentimestamps.org` — Peter Todd
- `finney.calendar.eternitywall.com` — Eternity Wall

実装は異なるオペレーターを使用しても **構いません(MAY)**。証人多様性(witness-diversity)ポリシー(独立したオペレーター ≥2)は規範的(normative)です。特定のオペレーターはそうではありません。

## 5. コンテンツアドレッシングによるミラーリング

公開ページと事実ファイルを含む `site/` ディレクトリは IPFS に追加する **べきです(SHOULD)**。

```
ipfs add -r --cid-version=1 -Q site/
```

得られる CIDv1(dag-pb コーデック)が、その公開物のコンテンツアドレスです。これは `site/` のバイト内容から決定論的に導かれます。`site/` 配下のいずれかのファイルへの変更は、異なる CID を生成します。

発行者のローカルノードを超えた追加のサービスへのピン留めは、永続性のために強く推奨されます。本仕様は特定のサービスを必須としません。

## 6. バージョニング

本プロトコルは **バージョンレベルで追記専用(append-only)** です。すべての変更には新しい封緘済みバージョンが必要です。

- 新しい(または変更なしの)仕様改訂
- 新しい事実集合。先行バージョンから変更なしの事実を継承して **構いません(MAY)**
- 新しい Merkle ルート
- 新しい OpenTimestamps レシート
- 新しい IPFS CID

すべての先行バージョンは、バージョン付き URL で公開され続けます。v(N) のいずれかの事実への変更には v(N+1) が必要であり、両バージョンが可視かつ再検証可能な状態に保たれます。

## 7. 検証

任意の第三者は、以下によって公開物の完全性を検証できます。

1. 事実ファイルをダウンロードし、各事実の SHA-256 ハッシュを再計算する。
2. §3 に従って Merkle ルートを再計算し、公開されたルートと比較する。
3. 少なくとも 1 つのカレンダーの完全な証明について、OpenTimestamps レシートを Bitcoin ブロックチェーンに照らして検証する。
4. `ipfs add -r --cid-version=1` で IPFS の CID を独立に再計算し、公開された CID と比較する。

`lib/verify.py` はステップ 1〜3 を、Python 標準ライブラリと `ots` クライアント以外の外部依存なしで実行します。ステップ 4 には `ipfs` クライアントが必要です。

## 8. 拒否ゲート(Refusal gate)

公開物は、以下のいずれかに該当する事実を含んで **はなりません(MUST NOT)**。

- 名指しされることに同意していない、私人(private third party)を特定するもの。
- 仮説を確認(confirmation)として提示するもの。
- 検証可能な命名/宣言イベントまたは暗号的アーティファクトに還元できないもの。
- 公開時に予防可能な死亡の合理的に予想される増加(reasonable expected increase in preventable deaths)をもたらすもの。

4 番目のルールが荷重を担う(load-bearing)ルールです。これは発行者による人間レビューのステップで施行されます。拒否ゲートにはプロトコルレベルでの施行はありません。プロトコルは与えられたバイト列をそのまま公開します。ゲートに対する責任は発行者にあります。

## 9. 限界の正直な表明

- **検出可能性であって、不可能性ではない。** 本プロトコルは改ざんを検出可能にするのであって、不可能にするのではありません。十分な意志のある攻撃者は、依然として別の公開物に差し替え、一部の読者にそれが原本であると信じ込ませることができます。プロトコルの価値は、その差し替えが、チェックを実行する誰にとっても暗号的に立証可能になることにあります。
- **虚偽の公開に対する防御はない。** 発行者が「X が起こった」という事実を封緘し、X が実際には起こっていなかった場合、プロトコルは真実を封緘するのとまったく同じ忠実さで虚偽の主張を封緘します。プロトコルが証明するのは何が *公開された* かであって、何が *真実である* かではありません。
- **単一当事者署名。** デプロイ済みの MYRIAM の v1 は、単一の署名主体(signing principal)を用いています。マルチパーティ署名は将来課題です。
- **ピン留めの永続性。** 発行者のノードのみへの IPFS ピン留めが提供するのは、「私のノートパソコンが起動している間 + 機会的なゲートウェイキャッシング」だけの可用性です。長期的な永続性には、独立したサービスをまたぐ冗長なピンが必要です。

これらの限界は、いかなる読者も本プロトコルがそれが提供する以上のものを提供すると仮定しないように、ここに明示されています。
