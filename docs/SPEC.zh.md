# MYRIAM-kit 协议规范

*中文翻译 — 英文原文见 [SPEC.md](SPEC.md)*

## 0. 目的

本文档规定了一套最小协议,用于以可察觉篡改、可密码学验证、且在任一托管服务被下架后仍可存续的形式,发布一小组文本事实。

它是首次在 [disclosure.host](https://disclosure.host) 以"MYRIAM v1"名义部署的协议的泛化。

## 1. 非目标

- 机密存储。所有内容按构造皆为公开。
- 大规模数据集。一旦超过约 20 项事实,本协议在形态上即不再适用。
- 实时发布。每次发布都是一次离散的密封事件。
- 撤回。本协议在版本层面是只追加的。

## 2. 事实模型

一项 FACT 是一个 UTF-8 文本文件,具备如下属性:

- 文件名格式:`NN_short_name.txt`,其中 NN 是从 01 开始的零填充两位编号。
- 规范形式:字节按原样保留,不做归一化。
- 每项事实的哈希:`SHA-256(file_bytes)` 的小写十六进制表示。

事实 SET(集合)是 `facts/` 目录下所有符合文件名格式的文件的有序列表,按字典序排序。

约束:每项事实的正文 SHOULD(应当)被表述为可归属于特定一方的*命名事件*或*声明事件*。本体论主张("月亮是岩石")会削弱可验证性门槛;唯有发布事件可被验证,主张本身则不可。正确形式为"X 声明 Y";错误形式为"Y 为真"。

## 3. Merkle 根

采用 Bitcoin 风格,与已部署的 MYRIAM 协议 v0 一致:

1. 叶子:按事实集合顺序排列的每项事实的 SHA-256 哈希。
2. 若某一层节点数为奇数,则复制最后一个节点。
3. 对相邻节点做成对哈希:`parent = SHA-256(left_bytes || right_bytes)`,其中 `left_bytes` 与 `right_bytes` 为原始的 32 字节哈希值(而非其十六进制表示)。
4. 重复,直至只剩下一个节点。该节点即为 Merkle 根。

该根以小写十六进制形式发布,末尾不带换行符。

### 3.1 测试向量

| Vector | Leaves (UTF-8 bytes hashed) | Expected root (hex) |
|---|---|---|
| A | `"a"` | `ca978112ca1bbdcafac231b39a23dc4da786eff8147c4e72b9807785afee48bb` |
| B | `"a"`, `"b"` | `e5a01fee14e0ed5c48714f22180f25ad8365b53f9779f79dc4a3d7e93963f94a` |
| C | `"a"`, `"b"`, `"c"` | `d31a37ef6ac14a2db1470c4316beb5592e6afd4465022339adafda76a18ffabe` |

任何合规实现 MUST(必须)能复现上述向量。它们由实际的 `lib/build.py` 导出,可使用该脚本重新生成。

## 4. Bitcoin 承诺

Merkle 根通过 [OpenTimestamps](https://opentimestamps.org) 提交到 Bitcoin 区块链。默认策略是:

> 至少向两个处于独立行政控制之下的日历运营方发起戳记。

满足该策略的收据可由任何能运行 `ots verify <receipt>.ots` 并到达任一日历的升级路径的人验证。

`lib/seal.sh` 中默认附带的运营方组合为:

- `alice.btc.calendar.opentimestamps.org` —— Peter Todd
- `finney.calendar.eternitywall.com` —— Eternity Wall

实现 MAY(可以)使用不同的运营方。见证多样性策略(≥2 个独立运营方)是规范性要求;具体运营方则不是。

## 5. 内容寻址镜像

包含已发布页面和事实文件的 `site/` 目录 SHOULD(应当)加入 IPFS:

```
ipfs add -r --cid-version=1 -Q site/
```

所得到的 CIDv1(dag-pb 编码)即该次发布的内容地址。它由 `site/` 的字节内容确定性地导出。`site/` 下任一文件的任何改动都会产生不同的 CID。

强烈建议将其 pin 到发布者本地节点之外的额外服务上以增强持久性;本规范不指定具体服务。

## 6. 版本管理

本协议在**版本层面是只追加的**。每一次变更都需要新的密封版本:

- 一份新的(或未变更的)规范修订
- 一组新的事实集,其中 MAY(可以)从先前版本继承未变更的事实
- 一个新的 Merkle 根
- 一份新的 OpenTimestamps 收据
- 一个新的 IPFS CID

所有先前版本仍保留在带版本号的 URL 上发布。对 v(N) 中任一事实的修改要求发布 v(N+1),且两个版本均可见、均可再次验证。

## 7. 验证

任何第三方都可以通过以下方式验证一次发布的完整性:

1. 下载事实文件,重新计算每项事实的 SHA-256 哈希。
2. 按 §3 重新计算 Merkle 根,并与所发布的根进行对比。
3. 针对至少一个日历的完整证明,将 OpenTimestamps 收据对照 Bitcoin 区块链进行验证。
4. 通过 `ipfs add -r --cid-version=1` 独立地重新计算 IPFS CID,并与所发布的 CID 进行对比。

`lib/verify.py` 执行第 1–3 步,除 Python 标准库与 `ots` 客户端外,不依赖任何外部组件。第 4 步需要 `ipfs` 客户端。

## 8. 拒发门槛

发布 MUST NOT(不得)包含如下事实:

- 指认未曾同意被指名的私人第三方。
- 将假设当作已证实呈现。
- 无法归约为可验证的命名/声明事件或密码学产物。
- 发布后存在合理可预期的可避免死亡增量。

第四条是承重条款。它由发布者在人工审查环节执行。本协议层面对拒发门槛不设强制;协议会发布交给它的任何字节。门槛的把守责任落在发布者身上。

## 9. 关于局限的诚实声明

- **可察觉,而非不可能。**本协议使篡改可被察觉,但并非不可能发生。坚定的攻击者仍可替换出不同的发布物,并使一部分读者相信那就是原版。本协议的价值在于:这种替换对任何愿意运行核验的人而言,都将变得可在密码学意义上被证明。
- **无法防御发布不实之事。**如果发布者密封了一项写有"X 发生过"的事实而 X 并未发生,本协议会以与密封真实事实同等的保真度密封该不实主张。本协议证明的是*被发布的内容*,而非*为真的内容*。
- **单方签名。**已部署的 MYRIAM v1 使用单一签名主体。多方签名是未来工作。
- **Pin 的持久性。**仅在发布者节点上 pin IPFS,所提供的只是"我的笔记本电脑开机时可用 + 偶发性网关缓存"。长期持久性需要跨多个独立服务的冗余 pin。

此处明确列出这些局限,是为了让任何读者都不至于误以为本协议能提供超出其实际能力的东西。
