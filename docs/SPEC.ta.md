# MYRIAM-kit நெறிமுறை விவரக்குறிப்பு

*தமிழ் மொழிபெயர்ப்பு — மூல ஆங்கில பதிப்புக்கு, [SPEC.md](SPEC.md) காண்க*

## 0. நோக்கம்

இந்த ஆவணம், ஒரு சிறிய தொகுப்பு உரை உண்மைகளை சேதம்-வெளிப்படையான, கிரிப்டோகிராஃபிக் ரீதியில் சரிபார்க்கக்கூடிய வடிவத்தில் வெளியிடுவதற்கான ஒரு குறைந்தபட்ச நெறிமுறையை விவரிக்கிறது, இது எந்தவொரு ஒற்றை ஹோஸ்டிங் வழங்குநரின் அகற்றத்திற்குப் பிறகும் தொடர்ந்து இருக்கும்.

இது [disclosure.host](https://disclosure.host) தளத்தில் "MYRIAM v1" என்ற பெயரில் முதன்முதலில் பயன்படுத்தப்பட்ட நெறிமுறையின் பொதுமயமாக்கம் (generalization) ஆகும்.

## 1. நோக்கம்-அல்லாதவை

- இரகசிய சேமிப்பு. கட்டுமானத்தினாலேயே அனைத்தும் பொதுவானவை.
- பெரிய தரவுத்தொகுப்புகள். ~20 உண்மைகளுக்கு மேல், நெறிமுறை தவறான வடிவம்.
- நிகழ்நேர வெளியீடு. ஒவ்வொரு வெளியீடும் ஒரு தனித்துவமான முத்திரை வைக்கும் நிகழ்வு.
- திரும்பப் பெறுதல். நெறிமுறை பதிப்பு-நிலையில் சேர்க்க-மட்டுமே (append-only) ஆகும்.

## 2. உண்மை மாதிரி

ஒரு FACT என்பது பின்வரும் பண்புகளுடனான UTF-8 உரை கோப்பு:

- கோப்புப் பெயர் வடிவம்: `NN_short_name.txt`, இங்கு NN என்பது 01-இலிருந்து தொடங்கும் சுழியால் நிரப்பப்பட்ட இரு-இலக்க அட்டவணை.
- நியமன வடிவம்: எழுதப்பட்டபடி பைட்டுகள், எந்த இயல்பாக்கமும் இல்லை.
- ஒவ்வொரு-உண்மை ஹாஷ்: `SHA-256(file_bytes)`-இன் சிற்றெழுத்து ஹெக்ஸ்.

உண்மை SET என்பது `facts/`-இன் கீழ் வடிவத்துடன் பொருந்தும் அனைத்து கோப்புகளின் வரிசைப்படுத்தப்பட்ட பட்டியல், அகராதி வரிசையில் (lexicographically) வரிசைப்படுத்தப்பட்டது.

கட்டுப்பாடு: ஒவ்வொரு உண்மை உடலும் ஒரு குறிப்பிட்ட தரப்பினருக்கு கூறக்கூடிய ஒரு *பெயரிடல் நிகழ்வு* அல்லது *அறிவிப்பு நிகழ்வு* ஆக கூறப்பட வேண்டும். ஆன்டாலஜிக்கல் கூற்றுகள் ("நிலா பாறை") சரிபார்ப்பு வாயிலை குறைமதிப்பிற்கு உட்படுத்துகின்றன; வெளியீட்டு நிகழ்வு மட்டுமே சரிபார்க்கக்கூடியது, கூற்று அல்ல. "X என்பது Y-ஐ கூறியுள்ளது" சரியான வடிவம். "Y உண்மை" அல்ல.

## 3. Merkle ரூட்

Bitcoin-பாணி, பயன்படுத்தப்பட்ட MYRIAM நெறிமுறையின் v0-இல் உள்ளபடி:

1. இலைகள்: உண்மை-தொகுப்பு வரிசையில் ஒவ்வொரு-உண்மை SHA-256 ஹாஷ்கள்.
2. ஒரு நிலையில் ஒற்றை எண்ணிக்கையிலான முனைகள் இருந்தால், கடைசி முனையை நகலெடுக்கவும்.
3. அடுத்தடுத்த முனைகளை ஜோடி-ஹாஷ் செய்யவும்: `parent = SHA-256(left_bytes || right_bytes)`,
   இங்கு `left_bytes` மற்றும் `right_bytes` என்பவை மூல 32-பைட் ஹாஷ் மதிப்புகள்
   (அவற்றின் ஹெக்ஸ் பிரதிநிதித்துவங்கள் அல்ல).
4. ஒரு முனை மட்டும் இருக்கும் வரை திரும்பவும். அதுவே Merkle ரூட்.

ரூட் சிற்றெழுத்து ஹெக்ஸாக, பின்தொடரும் புதிய வரி இல்லாமல் வெளியிடப்படுகிறது.

### 3.1 சோதனை வெக்டர்கள்

| Vector | Leaves (UTF-8 bytes hashed) | Expected root (hex) |
|---|---|---|
| A | `"a"` | `ca978112ca1bbdcafac231b39a23dc4da786eff8147c4e72b9807785afee48bb` |
| B | `"a"`, `"b"` | `e5a01fee14e0ed5c48714f22180f25ad8365b53f9779f79dc4a3d7e93963f94a` |
| C | `"a"`, `"b"`, `"c"` | `d31a37ef6ac14a2db1470c4316beb5592e6afd4465022339adafda76a18ffabe` |

எந்தவொரு பொருந்தும் செயல்படுத்தலும் (compliant implementation) இந்த வெக்டர்களை மீளுருவாக்க வேண்டும். அவை உண்மையான `lib/build.py`-இலிருந்து பெறப்பட்டவை, மேலும் அந்த ஸ்கிரிப்ட் மூலம் மீண்டும் உருவாக்கப்படலாம்.

## 4. Bitcoin உறுதிமொழி

Merkle ரூட் [OpenTimestamps](https://opentimestamps.org) வழியாக Bitcoin பிளாக்செயினுக்கு உறுதிமொழியளிக்கப்படுகிறது. இயல்புநிலை கொள்கை:

> சுயாதீன நிர்வாக கட்டுப்பாட்டின் கீழ் குறைந்தபட்சம் இரண்டு காலெண்டர் ஆபரேட்டர்களுக்கு எதிராக முத்திரையிடவும்.

இந்த கொள்கையை திருப்தி செய்யும் ஒரு ரசீது, `ots verify <receipt>.ots`-ஐ இயக்கி இரு காலெண்டர்களில் ஒன்றின் மேம்படுத்தல் பாதையை அடையக்கூடிய எந்தவொரு தரப்பினராலும் சரிபார்க்கக்கூடியது.

`lib/seal.sh`-இல் வழங்கப்படும் இயல்புநிலை ஆபரேட்டர் ஜோடி:

- `alice.btc.calendar.opentimestamps.org` — Peter Todd
- `finney.calendar.eternitywall.com` — Eternity Wall

செயல்படுத்தல்கள் வேறுபட்ட ஆபரேட்டர்களை பயன்படுத்தலாம். சாட்சி-பன்முகத்தன்மை கொள்கை (≥2 சுயாதீன ஆபரேட்டர்கள்) நியதிப்படியான (normative); குறிப்பிட்ட ஆபரேட்டர்கள் அல்ல.

## 5. உள்ளடக்கம்-முகவரியிடப்பட்ட பிரதிநிதியாக்கம் (mirroring)

வெளியிடப்பட்ட பக்கம் மற்றும் உண்மை கோப்புகளை உள்ளடக்கிய `site/` கோப்பகம் IPFS-இல் சேர்க்கப்பட வேண்டும்:

```
ipfs add -r --cid-version=1 -Q site/
```

இதன் விளைவாக உருவாகும் CIDv1 (dag-pb codec) வெளியீட்டின் உள்ளடக்க முகவரி ஆகும். இது `site/`-இன் பைட் உள்ளடக்கத்திலிருந்து தீர்மானகரமானது. `site/`-இன் கீழ் எந்த கோப்பிற்கும் ஏற்படும் எந்த மாற்றமும் வேறுபட்ட CID-ஐ உருவாக்கும்.

ஆயுள் நிலைத்தன்மைக்காக வெளியீட்டாளரின் உள்ளூர் நோடைத் தாண்டிய கூடுதல் சேவைகளுக்கு பின் செய்வது வலுவாக பரிந்துரைக்கப்படுகிறது; இந்த விவரக்குறிப்பு குறிப்பிட்ட சேவைகளை கட்டாயப்படுத்தவில்லை.

## 6. பதிப்பாக்கம் (Versioning)

நெறிமுறை **பதிப்பு-நிலையில் சேர்க்க-மட்டுமே (append-only)** ஆகும். ஒவ்வொரு மாற்றத்திற்கும் ஒரு புதிய முத்திரை வைக்கப்பட்ட பதிப்பு தேவைப்படுகிறது:

- ஒரு புதிய (அல்லது மாற்றப்படாத) விவரக்குறிப்பு திருத்தம்
- ஒரு புதிய உண்மை தொகுப்பு, இது முந்தைய பதிப்புகளிலிருந்து உண்மைகளை மாற்றமின்றி வாரிசாகப் பெறலாம்
- ஒரு புதிய Merkle ரூட்
- ஒரு புதிய OpenTimestamps ரசீது
- ஒரு புதிய IPFS CID

அனைத்து முந்தைய பதிப்புகளும் பதிப்புள்ள URL-களில் வெளியிடப்பட்டபடியே இருக்கின்றன. எந்தவொரு v(N) உண்மையின் மாற்றத்திற்கும் v(N+1) தேவைப்படுகிறது, இரண்டு பதிப்புகளும் தெரியக்கூடியதாகவும் மீள்-சரிபார்க்கக்கூடியதாகவும் இருக்கும்.

## 7. சரிபார்ப்பு

எந்தவொரு மூன்றாம் தரப்பினரும் ஒரு வெளியீட்டின் நேர்மையை இவ்வாறு சரிபார்க்கலாம்:

1. உண்மை கோப்புகளை பதிவிறக்கி ஒவ்வொரு-உண்மை SHA-256 ஹாஷ்களை மீண்டும் கணக்கிடவும்.
2. §3-இன் படி Merkle ரூட்டை மீண்டும் கணக்கிட்டு வெளியிடப்பட்ட ரூட்டுடன் ஒப்பிடவும்.
3. குறைந்தபட்சம் ஒரு காலெண்டரின் முழு சான்றளிப்புக்கு Bitcoin பிளாக்செயினுக்கு எதிராக OpenTimestamps ரசீதை சரிபார்க்கவும்.
4. `ipfs add -r --cid-version=1` வழியாக IPFS CID-ஐ சுயாதீனமாக மீண்டும் கணக்கிட்டு வெளியிடப்பட்ட CID-உடன் ஒப்பிடவும்.

Python stdlib மற்றும் `ots` கிளையன்ட்-ஐத் தாண்டி வெளிப்புற சார்புகள் எதுவும் இல்லாமல், படிகள் 1–3-ஐ `lib/verify.py` செய்கிறது. படி 4-க்கு `ipfs` கிளையன்ட் தேவை.

## 8. மறுப்பு வாயில் (Refusal gate)

ஒரு வெளியீடு பின்வரும் ஒரு உண்மையை உள்ளடக்க கூடாது:

- பெயரிடப்படுவதற்கு சம்மதிக்காத ஒரு தனியார் மூன்றாம் தரப்பினரை அடையாளம் காண்பது.
- ஒரு கருதுகோளை உறுதிப்பாடாக வழங்குவது.
- ஒரு சரிபார்க்கக்கூடிய பெயரிடல்/அறிவிப்பு நிகழ்வாக அல்லது ஒரு கிரிப்டோகிராஃபிக் கலைப்பொருளாக குறைக்க முடியாதது.
- வெளியீட்டின் போது தடுக்கக்கூடிய மரணங்களில் நியாயமான எதிர்பார்க்கப்படும் அதிகரிப்பு உள்ளது.

நான்காவது விதி சுமை-தாங்கும் ஒன்று. இது வெளியீட்டாளரால் மனித மறுபார்வை படியில் அமலாக்கப்படுகிறது. மறுப்பு வாயிலுக்கு நெறிமுறை-நிலை அமலாக்கம் எதுவும் இல்லை; நெறிமுறை அதற்கு கொடுக்கப்படும் எந்த பைட்டுகளையும் வெளியிடுகிறது. வாயிலுக்கு வெளியீட்டாளர் பொறுப்பு.

## 9. வரம்புகளின் நேர்மையான கூற்று

- **கண்டறியும் தன்மை, இயலாமை அல்ல.** நெறிமுறை சேதத்தை இயலாததாக அல்ல, கண்டறியக்கூடியதாக ஆக்குகிறது. ஒரு தீர்மானமான தாக்குதலாளர் இன்னும் வேறுபட்ட வெளியீட்டை மாற்றியமைத்து சில வாசகர்களை அதுவே மூலம் என்று நம்ப வைக்க முடியும். நெறிமுறையின் மதிப்பு, மாற்றீடு சரிபார்ப்பை இயக்கும் எவருக்கும் கிரிப்டோகிராஃபிக் ரீதியாக நிரூபிக்கக்கூடியதாக மாறுகிறது என்பதே.
- **பொய் வெளியீட்டிற்கு எதிராக பாதுகாப்பு இல்லை.** வெளியீட்டாளர் "X நடந்தது" என்று கூறும் ஒரு உண்மையை முத்திரை வைத்து, X நடக்கவில்லை என்றால், நெறிமுறை உண்மையானவற்றை முத்திரை வைப்பதைப் போலவே அதே நம்பகத்தன்மையுடன் தவறான கூற்றையும் முத்திரை வைக்கிறது. நெறிமுறை *வெளியிடப்பட்டதை* சான்றளிக்கிறது, *உண்மையானதை* அல்ல.
- **ஒற்றை-தரப்பு கையொப்பமிடல்.** பயன்படுத்தப்பட்ட MYRIAM-இன் v1 ஒற்றை கையொப்ப முதன்மையை பயன்படுத்துகிறது. பல-தரப்பு கையொப்பமிடல் எதிர்கால வேலை.
- **பின்னிங் நிலைத்தன்மை.** வெளியீட்டாளரின் நோடில் மட்டும் IPFS பின்னிங் "என் லேப்டாப் இயங்கும்போது + வாய்ப்புசார் கேட்வே கேச்சிங்" மட்டுமே வழங்குகிறது. நீண்டகால நிலைத்தன்மைக்கு சுயாதீன சேவைகளில் பல மிகுந்த பின்கள் தேவைப்படுகின்றன.

இந்த வரம்புகள் இங்கே கூறப்பட்டுள்ளன, எந்த வாசகரும் நெறிமுறை அதைவிட அதிகமாக வழங்குகிறது என்று கருதாமல் இருக்க.
